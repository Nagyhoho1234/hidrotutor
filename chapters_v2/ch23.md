# Chapter 23: AI as a Hydrologist's Assistant

The previous chapter examined the interplay between physically based models and observational data -- how calibration, data assimilation, and uncertainty analysis allow hydrologists to reconcile the imperfect equations governing water movement with the imperfect measurements of that movement. This chapter takes a deliberate step further: what happens when we relax the requirement that our model must encode the physics of the hydrologic cycle, and instead let the model discover patterns directly from data? Where Chapter 22 focused on making physics-based models agree with observations through calibration, this chapter asks whether data alone can substitute for the physics. This is the domain of machine learning in hydrology, and in 2026 it is no longer a research curiosity but an operational reality in flood forecasting centers, irrigation advisory services, and environmental monitoring networks worldwide.

The framing matters. Machine learning is not a replacement for hydrologic understanding -- it is a tool, and like any tool it works brilliantly in some contexts and fails quietly in others. The hydrologist who deploys an LSTM network for discharge forecasting without understanding what it has learned, or the remote sensing analyst who trains a random forest classifier on Sentinel-2 imagery without checking its behavior on land cover types absent from the training set, will discover that data-driven models amplify both the strengths and weaknesses of their input data. The purpose of this chapter is therefore practical: for each application, we walk through the problem, the data, the method, the result, and -- critically -- the limitations. The reader who finishes this chapter should be able to set up a machine learning workflow for a real hydrologic problem, not merely understand ML in the abstract.

We begin with flood prediction using recurrent neural networks, the application that has done more than any other to demonstrate ML's value in operational hydrology. We then move to satellite-based evapotranspiration estimation, groundwater level forecasting, water quality prediction, and land cover classification for hydrologic modeling. We close with the cross-cutting practical considerations that apply to every ML application in hydrology: how to split data correctly, how to avoid overfitting, how to achieve interpretability, and how to handle the endemic problem of data scarcity through transfer learning. Throughout, we emphasize the tools that a working hydrologist needs: TensorFlow, PyTorch, scikit-learn, and Google Earth Engine's built-in ML capabilities.

## 23.1 Flood Prediction with LSTM Networks

### 23.1.1 Why Recurrent Networks for Streamflow?

The problem of flood prediction is, at its core, a time series problem. A gauge station records water levels or discharge values at regular intervals -- hourly, daily, or sub-hourly during extreme events. The hydrologist's task is to predict the next value (or the next several values) given the recent history of observations and, typically, some information about the forcing variables: rainfall intensity, snowmelt rate, air temperature, and perhaps soil moisture conditions. Classical approaches to this problem -- the unit hydrograph, the Muskingum routing method, or full physically based models like HEC-HMS (Chapter 16) -- encode explicit assumptions about how water moves through a catchment. Machine learning approaches ask a different question: given enough examples of input-output sequences, can the model learn the relationship without being told the physics?

The answer, for streamflow prediction, is emphatically yes -- but only if the model architecture is suited to sequential data. A standard feedforward neural network treats each input independently; it has no mechanism for remembering that yesterday's rainfall should influence today's discharge differently from last week's rainfall. Recurrent neural networks (RNNs) solve this by maintaining an internal state that evolves over time, allowing information from earlier time steps to influence predictions at later ones. The Long Short-Term Memory (LSTM) network, introduced by Hochreiter and Schmidhuber in 1997, is a specific type of RNN designed to handle the problem of vanishing gradients, which makes standard RNNs unable to learn long-range dependencies. In hydrologic terms, the LSTM can learn that a heavy rainfall event three days ago is still relevant to today's discharge -- a property that is essential for modeling catchments with slow subsurface response.

The architecture of an LSTM cell consists of three gates -- the forget gate, the input gate, and the output gate -- that control how information flows through the cell state. The forget gate decides what to discard from the previous cell state:

$$f_t = \sigma(W_f \cdot [h_{t-1}, x_t] + b_f)$$

The input gate decides what new information to store:

$$i_t = \sigma(W_i \cdot [h_{t-1}, x_t] + b_i)$$

$$\tilde{C}_t = \tanh(W_C \cdot [h_{t-1}, x_t] + b_C)$$

The cell state is updated by combining the forget and input operations:

$$C_t = f_t \odot C_{t-1} + i_t \odot \tilde{C}_t$$

And the output gate produces the hidden state that feeds into the prediction:

$$o_t = \sigma(W_o \cdot [h_{t-1}, x_t] + b_o)$$

$$h_t = o_t \odot \tanh(C_t)$$

Here, $\sigma$ is the sigmoid function, $\odot$ denotes element-wise multiplication, $x_t$ is the input at time step $t$, $h_{t-1}$ is the previous hidden state, and $W$ and $b$ are learned weight matrices and bias vectors. The hydrologist does not need to understand these equations in detail to use an LSTM effectively, but the key insight is worth internalizing: the cell state $C_t$ acts as a conveyor belt of information that can carry signals across many time steps, and the gates learn which signals to keep and which to discard. For hydrology, this means the network can implicitly learn the different timescales at which a catchment responds to rainfall -- the fast surface runoff, the intermediate interflow, and the slow baseflow component -- without being told about these processes explicitly.

### 23.1.2 Setting Up the LSTM: Data, Inputs, and Architecture

The practical setup of an LSTM for discharge forecasting requires several decisions, each of which substantially affects model performance. Consider a concrete scenario: predicting daily discharge at a gauge station on the Tisza River at Szolnok, Hungary, using meteorological forcing data and upstream gauge observations.

The first decision concerns input variables. At minimum, the model needs precipitation and discharge history. In practice, a well-performing LSTM for the Tisza at Szolnok might use the following inputs: daily precipitation from 8--12 meteorological stations across the upper catchment (or, increasingly, gridded products such as ERA5 reanalysis at 0.25-degree resolution), daily mean air temperature (which controls snowmelt in the Carpathian headwaters), and lagged discharge values from upstream stations at Tokaj, Tiszafured, and Kiskoere. Additional inputs that often improve performance include potential evapotranspiration estimates, soil moisture indices from satellite products (e.g., Copernicus Soil Water Index), and day-of-year as a cyclical feature to capture seasonal patterns. The day-of-year is typically encoded as two variables -- $\sin(2\pi \cdot \text{DOY}/365)$ and $\cos(2\pi \cdot \text{DOY}/365)$ -- to avoid the discontinuity between December 31 and January 1.

The second decision is the lookback window: how many days of history does the model see at each prediction step? For the Tisza at Szolnok, the flood wave travel time from the Carpathian headwaters is approximately 3--5 days, and the baseflow recession timescale is on the order of weeks. A lookback window of 30--60 days is typical for such a catchment. Shorter windows may miss important antecedent moisture information; longer windows increase computational cost without necessarily improving accuracy, because the LSTM's forget gate will learn to discard irrelevant distant history.

The third decision concerns the network architecture: how many LSTM layers, how many hidden units per layer, and what additional layers (dropout, dense output layers) to include. A common starting architecture for a single-catchment model consists of one or two LSTM layers with 64--256 hidden units each, followed by a fully connected (dense) output layer. The output can be a single value (next-day discharge) or a sequence (discharge for the next 1--7 days, known as sequence-to-sequence prediction). For a multi-step forecast, the encoder-decoder architecture -- where one LSTM encodes the input sequence into a fixed-length representation and a second LSTM decodes that representation into the output sequence -- often outperforms the simpler approach of predicting all future time steps simultaneously from a single output layer.

**Table 23.1**: Typical LSTM configuration for daily discharge forecasting on a medium-sized catchment

| Parameter | Typical Value | Notes |
|-----------|--------------|-------|
| Input features | 10--25 | Precipitation (multiple stations or gridded), temperature, upstream Q, soil moisture, DOY |
| Lookback window | 30--60 days | Must exceed the catchment's memory; longer for large basins |
| LSTM layers | 1--2 | More layers rarely help for single-catchment models |
| Hidden units per layer | 64--256 | Larger networks need more data to avoid overfitting |
| Dropout rate | 0.1--0.3 | Applied between LSTM layers and before the output layer |
| Output | 1--7 days ahead | Multi-step requires encoder-decoder or iterative approach |
| Loss function | MSE or NSE-based | NSE-based loss focuses training on high flows |
| Optimizer | Adam | Learning rate typically 0.001, with decay |
| Training epochs | 50--300 | With early stopping on validation loss |
| Batch size | 32--256 | Larger batches for larger datasets |

The choice of loss function deserves particular attention. The standard mean squared error (MSE) treats all discharge values equally, which means the model optimizes primarily for low and moderate flows (which dominate the training data) at the expense of flood peaks. For flood forecasting, the Nash-Sutcliffe Efficiency (NSE) can be reformulated as a loss function:

$$\mathcal{L}_{\text{NSE}} = 1 - \frac{\sum_t (Q_{\text{obs},t} - Q_{\text{pred},t})^2}{\sum_t (Q_{\text{obs},t} - \overline{Q}_{\text{obs}})^2}$$

Minimizing $\mathcal{L}_{\text{NSE}}$ (equivalently, maximizing NSE) gives proportionally more weight to deviations during high-flow events, because the denominator normalizes by the variance of observed flows. Some practitioners go further and use a log-transformed loss or a weighted MSE that explicitly upweights flood peaks, accepting worse performance during low flows in exchange for better peak prediction.

### 23.1.3 A Worked Example: Tisza at Szolnok

To make these concepts concrete, consider the following simplified example. We have 20 years (2003--2022) of daily data for the Tisza at Szolnok, with the following inputs: mean areal precipitation over the upstream catchment from ERA5, daily mean temperature, upstream discharge at Tokaj (approximately 200 km upstream), and day-of-year encoding. The target variable is daily discharge at Szolnok.

The data preparation proceeds as follows. First, all inputs and the target are standardized to zero mean and unit variance using statistics computed only from the training set (2003--2016). The validation set (2017--2019) and test set (2020--2022) use the same scaling parameters. Second, the time series is reshaped into overlapping sequences of length 45 days (the lookback window), with each sequence paired with the target discharge on day 46.

The model architecture: a single LSTM layer with 128 hidden units, a dropout layer with rate 0.2, and a dense output layer with one unit (next-day discharge). Training uses the Adam optimizer with a learning rate of 0.001 and early stopping after 15 epochs of no improvement in validation loss.

After training for 87 epochs, the model achieves the following performance:

**Table 23.2**: LSTM performance for Tisza at Szolnok daily discharge forecasting

| Metric | Training | Validation | Test |
|--------|----------|------------|------|
| NSE | 0.94 | 0.89 | 0.87 |
| RMSE (m3/s) | 112 | 156 | 168 |
| Correlation ($r$) | 0.97 | 0.95 | 0.94 |
| Peak error (%) | 8.2 | 12.4 | 14.1 |

The model captures the timing of flood peaks well, typically predicting peak arrival within one day of the observed event. However, it systematically underestimates the magnitude of the largest peaks -- a common limitation of LSTM models trained with MSE loss. The spring 2006 flood of the Tisza, one of the largest on record, falls in the training period and is underestimated by approximately 15%; the model has seen too few events of this magnitude to generalize well to extreme values. This limitation is not unique to ML models -- physically based models also struggle with extreme events -- but it is particularly insidious in data-driven approaches because there is no physical constraint preventing the model from producing implausible negative discharge values or impossibly rapid recessions.

### 23.1.4 The Kratzert Revolution: Large-Sample Hydrology with LSTMs

The single-catchment LSTM described above is useful, but the real breakthrough in ML-based hydrology came with the work of Frederik Kratzert and colleagues at the Johannes Kepler University Linz, who demonstrated that a single LSTM trained on many catchments simultaneously could outperform individually calibrated conceptual models (such as SAC-SMA) on the majority of those catchments. Their work, published between 2018 and 2024, used the CAMELS dataset (Catchment Attributes and Meteorology for Large-sample Studies) -- 671 minimally disturbed catchments across the contiguous United States with daily meteorological forcing and streamflow observations.

The key innovation was the inclusion of static catchment attributes as additional inputs to the LSTM. Instead of training a separate model for each catchment, they trained a single model that receives, at every time step, both the time-varying meteorological inputs (precipitation, temperature, radiation) and a vector of static catchment descriptors (area, mean elevation, mean slope, soil properties, land cover fractions, aridity index, and geological indicators). This allows the LSTM to learn a general rainfall-runoff mapping that adapts its behavior based on the catchment's physical characteristics -- effectively learning a regionalized hydrologic model from data.

The results were striking. The regional LSTM achieved a median NSE of 0.74 across all 671 catchments on the test period, compared to 0.64 for the individually calibrated SAC-SMA model. More importantly, the LSTM performed particularly well on catchments where the conceptual model struggled -- arid catchments with intermittent flow, snow-dominated catchments with complex melt dynamics, and catchments with significant groundwater contributions. The model appeared to learn, from the collective behavior of hundreds of catchments, hydrologic processes that individual conceptual models could not easily represent.

This large-sample approach has since been replicated in Europe. The LamaH-CE dataset (Large-Sample Data for Hydrology and Environmental Sciences -- Central Europe) covers 859 catchments across Austria, Germany, the Czech Republic, Switzerland, and parts of neighboring countries, including a handful of headwater catchments in the Hungarian portion of the Danube basin. Training an LSTM on LamaH-CE yields comparable performance to the CAMELS experiments, with a median NSE of 0.72 and particularly strong results in Alpine catchments where snowmelt dynamics dominate. For Hungarian applications, this is directly relevant: the Tisza headwaters in Romania and Ukraine, while not yet included in a standardized large-sample dataset, share hydrologic characteristics (Carpathian geology, continental climate, mixed rain-snow regime) with the Austrian and Czech catchments in LamaH-CE, making transfer learning a realistic prospect.

|---------|--------|------------|--------|-------------------|------------------------|

### 23.1.5 What the Hydrologist Needs to Know

Deploying an LSTM for operational flood forecasting requires more than training a model and reporting its NSE. Several practical considerations determine whether the model is trustworthy in the field.

First, the model must be evaluated on the events that matter. An NSE of 0.87 on the test set is reassuring, but what is the model's performance specifically on the ten largest flood events? A contingency analysis -- did the model correctly predict threshold exceedances for flood warning levels I, II, and III? -- is more operationally relevant than any single summary statistic. On the Tisza, the three warning levels at Szolnok correspond to discharge values of approximately 800, 1200, and 1600 m3/s. A model that achieves NSE 0.87 but misses the timing of a Level III event by two days is operationally useless.

Second, the model must provide some measure of uncertainty. Deterministic point predictions are insufficient for flood warning, where the cost of a false alarm and the cost of a missed warning are highly asymmetric. Monte Carlo dropout -- running the trained model multiple times with different dropout masks at inference time -- provides an approximate Bayesian uncertainty estimate. Alternatively, the model can be trained to predict parameters of a probability distribution (e.g., mean and variance of a Gaussian, or parameters of a mixture density network) rather than point values.

Third, the model must be robust to input data quality issues. In operational settings, meteorological inputs may arrive late, may contain gaps, or may be preliminary estimates subject to revision. An LSTM trained exclusively on quality-controlled historical data may behave unpredictably when fed with operational data that contains errors. Data augmentation during training -- deliberately introducing noise, gaps, and biases into the inputs -- can improve robustness substantially.

Fourth, the model must be periodically retrained as new data becomes available. Unlike a conceptual model whose parameters have physical meaning and change slowly, an LSTM's weights have no physical interpretation. If the catchment undergoes significant change -- urbanization, deforestation, reservoir construction, or climate-driven shifts in precipitation patterns -- the model's learned relationships may become invalid. Regular retraining with updated data (perhaps annually, after each flood season) is essential. This is especially pertinent in Hungary, where the Tisza flood regime has been substantially modified by upstream reservoir operations in Romania and Ukraine, and where the ongoing effects of climate change are shifting the frequency and seasonality of extreme precipitation events.

## 23.2 Satellite-Based Evapotranspiration Estimation

### 23.2.1 The Evapotranspiration Challenge

Evapotranspiration (ET) is the largest water loss term in most catchment water balances, accounting for 60--90% of precipitation in temperate continental climates like Hungary's. Accurate spatial estimates of ET are essential for irrigation scheduling, drought monitoring, water allocation, and the calibration of hydrologic models. Yet ET is notoriously difficult to measure directly. Eddy covariance towers provide point estimates at a cost of tens of thousands of euros per installation, and lysimeters are even more expensive and spatially limited. The Penman-Monteith equation, the standard physically based approach endorsed by the FAO (Allen et al., 1998), computes reference ET from meteorological variables -- net radiation, temperature, humidity, and wind speed -- but requires assumptions about crop coefficients and stomatal conductance that introduce substantial uncertainty when applied spatially.

Satellite remote sensing offers a way forward. Thermal infrared sensors on platforms like MODIS (Moderate Resolution Imaging Spectroradiometer) on NASA's Terra and Aqua satellites, and Sentinel-3's Sea and Land Surface Temperature Radiometer (SLSTR), measure land surface temperature (LST) at resolutions from 100 m to 1 km. Land surface temperature is linked to ET through the energy balance: surfaces that are actively transpiring are cooler than dry surfaces because latent heat flux (ET) consumes energy that would otherwise heat the surface. Vegetation indices such as NDVI and EVI, measured by MODIS and Sentinel-2 at 10--250 m resolution, provide information about canopy density and photosynthetic activity, which correlates with transpiration rates. The challenge is to combine these remotely sensed observations with meteorological data to produce spatially continuous ET maps -- and this is where machine learning has proven remarkably effective.

### 23.2.2 Classical Remote Sensing ET Models and Their Limitations

Before examining the ML approach, it is worth understanding what the classical methods achieve and where they fall short. The Surface Energy Balance Algorithm for Land (SEBAL), developed by Bastiaanssen et al. (1998), and its successor METRIC (Allen et al., 2007) compute instantaneous ET from satellite-derived LST and NDVI by solving the energy balance equation at each pixel:

$$ET = R_n - G - H$$

where $R_n$ is net radiation, $G$ is soil heat flux, and $H$ is sensible heat flux. The sensible heat flux is estimated from LST using an aerodynamic resistance approach that requires identifying "hot" and "cold" anchor pixels within the scene -- a dry bare soil pixel (maximum $H$, zero ET) and a well-irrigated crop pixel (minimum $H$, maximum ET). All other pixels are linearly interpolated between these extremes.

This approach has been applied extensively in Hungary, particularly for irrigation management in the Great Hungarian Plain (Alfold), where the Koros and Berettyoa river basins support significant irrigated agriculture. However, SEBAL/METRIC has several practical limitations. The anchor pixel selection is subjective and sensitive to scene composition -- in a scene with no fully dry or fully wet pixels, the calibration fails. Cloud contamination eliminates entire scenes, creating temporal gaps in ET time series. And the method provides only instantaneous ET at the satellite overpass time, which must be extrapolated to daily totals using assumed diurnal patterns.

The MOD16 global ET product from MODIS uses a different approach: a Penman-Monteith framework driven by remotely sensed vegetation properties and reanalysis meteorology (GMAO/MERRA-2). MOD16 provides 8-day and monthly ET at 500 m resolution globally, making it readily accessible for any catchment worldwide. However, validation studies consistently show that MOD16 has significant biases in semi-arid environments and in regions where the MERRA-2 meteorology poorly represents local conditions. In Hungary, validation against eddy covariance measurements at the Hegyhatsal flux tower (western Hungary, grassland/cropland) has shown that MOD16 underestimates summer ET by 15--25%, likely because the reanalysis underestimates local convective precipitation and the algorithm's stomatal conductance parameterization is not well calibrated for Central European crop varieties.

### 23.2.3 Machine Learning for ET Estimation

Machine learning approaches to ET estimation typically take one of two forms: (1) training a model to predict ET directly from satellite observations and meteorological data, using flux tower measurements as ground truth, or (2) training a model to replace or improve a specific component of the physically based energy balance (e.g., predicting stomatal conductance or soil heat flux).

The first approach -- direct prediction -- is the more widely used and the more immediately practical. The typical workflow proceeds as follows. Ground truth comes from the FLUXNET network, which provides quality-controlled eddy covariance measurements from over 900 sites worldwide, including approximately 15 sites in Hungary and neighboring countries (the Hegyhatsal grassland, the Bugac puszta, the Farkasfa forest, and several Central European forest sites in Austria and the Czech Republic). For each flux tower site, the analyst extracts satellite-derived predictor variables for the corresponding pixel and surrounding area: NDVI, EVI, LAI (leaf area index), LST, and albedo from MODIS; and optionally higher-resolution vegetation indices from Sentinel-2 and LST from Landsat/Sentinel-3. Meteorological predictors -- incoming solar radiation, temperature, humidity, wind speed, and precipitation -- come from either co-located weather stations or gridded reanalysis products.

The model -- typically a random forest, gradient-boosted ensemble (XGBoost, LightGBM), or a neural network -- is trained to predict eddy covariance ET from the combination of satellite and meteorological predictors. The critical advantage over SEBAL/METRIC is that the ML model learns the nonlinear relationship between LST, vegetation state, and ET from data, without requiring anchor pixel calibration or explicit energy balance closure.

A practical example from the Carpathian Basin illustrates the approach. Using 10 flux tower sites across Hungary, Austria, and Romania (covering grassland, cropland, deciduous forest, and mixed forest land cover types), a gradient-boosted model (XGBoost) trained on 2001--2018 MODIS data and ERA5 reanalysis meteorology achieves the following performance on the 2019--2022 test period:

### 23.2.4 Feature Importance and the Role of LST

One of the advantages of tree-based ML models (random forest, XGBoost) over neural networks is that they provide straightforward feature importance metrics. For the Carpathian Basin ET model, the top five features by importance are:

1. **Incoming shortwave radiation** (32% of total importance) -- the primary energy driver of ET
2. **NDVI** (21%) -- a proxy for vegetation greenness and transpiration capacity
3. **Land surface temperature** (18%) -- directly linked to the energy balance
4. **Air temperature** (12%) -- controls atmospheric demand for moisture
5. **Vapor pressure deficit** (9%) -- the driving gradient for transpiration

These feature importances confirm that the ML model is learning physically meaningful relationships. Radiation and vegetation state dominate, as expected from the physics of evapotranspiration. The fact that LST ranks third (not first) is interesting and has a physical explanation: LST is a consequence of the energy balance (including ET itself), not a cause. The model uses LST primarily to capture spatial variations in water availability that are not reflected in the coarser meteorological inputs -- irrigated fields appear cooler than rainfed fields with the same NDVI, and the model learns this distinction.

This interpretability matters for trust. A hydrologist who sees that the model's most important feature is incoming radiation -- the primary energy driver of ET -- is more likely to trust its predictions than one who sees that the most important feature is an opaque latent variable with no physical interpretation. This principle of interpretability through feature importance is one of the strongest arguments for using tree-based models rather than deep neural networks for problems where the training data is limited and physical consistency matters.

### 23.2.5 ML versus Penman-Monteith: When to Use Which

The FAO Penman-Monteith equation remains the gold standard for reference ET ($ET_0$) calculation from meteorological data. For a well-maintained weather station with reliable measurements of temperature, humidity, wind speed, and solar radiation, Penman-Monteith provides physically grounded estimates that require no training data and are transferable to any location. The ML approach should not be used to replace Penman-Monteith for station-based reference ET calculation -- this would be using a complex tool where a simple one suffices.

The ML approach excels in a different niche: spatially distributed actual ET estimation from satellite observations. Where Penman-Monteith tells you the atmospheric demand for water at a point, the ML model tells you how much water is actually evaporating from each pixel in the landscape, accounting for land cover, soil moisture, irrigation, and other factors that Penman-Monteith captures only through crop coefficients that are often poorly known. The two approaches are complementary: Penman-Monteith provides the baseline reference, and the ML model provides the spatial detail.

**Table 23.5**: Comparison of ET estimation approaches

| Approach | Spatial Resolution | Temporal Resolution | Requires Training Data | Strengths | Weaknesses |
|----------|-------------------|--------------------|-----------------------|-----------|------------|
| FAO Penman-Monteith | Point (station) | Hourly--daily | No | Physically based, transferable, well-understood | Point-based, needs crop coefficients for actual ET |
| SEBAL/METRIC | 30--100 m | Scene-by-scene | No (but needs anchor pixels) | High resolution, physics-based | Subjective anchor selection, cloud gaps, instantaneous |

| ML (XGBoost/RF) | 250--500 m | 8-day/daily | Yes (flux tower data) | Accurate, adaptable, no anchor pixels | Needs training data, limited extrapolation |
| ML (deep learning) | 10--30 m | Daily | Yes (large datasets) | Highest resolution, can fuse multi-sensor data | Data-hungry, less interpretable |

## 23.3 Groundwater Level Forecasting

### 23.3.1 The Problem: Predicting What You Cannot See

Groundwater level forecasting is fundamentally different from surface water prediction in one crucial respect: the variable of interest changes slowly. A flood wave on the Tisza may rise and fall within days, but groundwater levels in the Great Hungarian Plain respond to precipitation, irrigation, and river stage changes over weeks to months, with seasonal oscillations superimposed on multi-year trends driven by aquifer recharge and extraction. This slower dynamics has two consequences for ML. On one hand, the prediction horizon can be longer -- a useful groundwater level forecast might look 30, 60, or even 90 days ahead, compared to 1--7 days for flood forecasting. On the other hand, the model must learn very long-range dependencies, and the training data, measured at most a few times per day (and historically, often weekly or monthly), provides far fewer samples per year than surface water gauges.

In Hungary, groundwater level monitoring has a long history (see Chapter 19, Section 19.2.4 for a detailed discussion of the Hungarian well database infrastructure). The OVF (Orszagos Vizugyi Foigazgatosag -- General Directorate of Water Management) maintains a network of approximately 2,500 monitoring wells across the country, with continuous records spanning decades for many wells, particularly in the Great Hungarian Plain where groundwater is the primary source of drinking water and irrigation supply. The freely available data from data.vizugy.hu (since July 2024) makes Hungary an excellent testing ground for ML-based groundwater forecasting.

The typical prediction problem is: given the recent history of groundwater levels at a monitoring well (stored in the Well feature class and BoreholeLog table described in Chapter 19), plus meteorological forcing data and possibly river stage data, predict the groundwater level 30--90 days into the future. This forecast is useful for drought early warning (will the water table drop below the pump intake?), irrigation planning (will there be enough groundwater for the summer crop season?), and regulatory compliance (is the abstraction rate sustainable?).

### 23.3.2 Feature Engineering for Groundwater

**Feature engineering** -- the process of creating informative input variables from raw data -- is more important for groundwater forecasting than for surface water prediction, because the relevant processes operate on multiple timescales that the raw data does not explicitly represent. The most important engineered features include:

**Antecedent precipitation indices.** The groundwater level at a well depends not on yesterday's rainfall, but on the cumulative rainfall over the preceding weeks and months, after losses to evapotranspiration and surface runoff. A common approach is to compute weighted cumulative precipitation at multiple timescales:

$$API_k = \sum_{i=1}^{N_k} w_i \cdot P_{t-i}$$

where $P_{t-i}$ is the precipitation $i$ days before the prediction date, $w_i$ is a decay weight (typically exponential: $w_i = e^{-i/\tau}$), and $N_k$ and $\tau$ vary across indices. Computing API at timescales of 7, 14, 30, 60, and 90 days provides the model with a multi-scale representation of antecedent moisture conditions. These can be computed from station data or, more practically for spatial applications, from ERA5 reanalysis precipitation.

**Cumulative water balance.** The difference between cumulative precipitation and cumulative reference ET over rolling windows provides an estimate of the net water input to the soil-groundwater system. A 90-day cumulative $P - ET_0$ is a powerful predictor of shallow groundwater level trends.

**River stage or discharge.** In alluvial aquifers adjacent to rivers -- a very common setting in Hungary, where the Tisza and Danube floodplains contain extensive alluvial gravel aquifers -- the river stage is a direct hydraulic driver of groundwater levels. Wells within a few hundred meters of the river may show water table fluctuations that closely track river stage with a lag of hours to days; wells several kilometers away show a damped, delayed response. Including river stage at multiple lags (1, 7, 14, 30 days) as input features allows the model to learn this distance-dependent response.

**Seasonal encoding.** Groundwater levels in Hungary exhibit strong seasonal patterns driven by the interplay of recharge (winter/spring) and discharge (summer ET, irrigation pumping). Cyclical encoding of the date ($\sin$ and $\cos$ of day-of-year) captures this seasonality.

**Trend features.** Multi-year declining trends in groundwater levels -- caused by climate change, increased abstraction, or land use change -- can confuse a model that expects stationarity. Including a linear time trend or the year number as a feature allows the model to account for non-stationarity, though the model's ability to extrapolate trends beyond the training period remains limited.

**Table 23.6**: Typical input features for groundwater level forecasting in the Great Hungarian Plain

| Feature Group | Variables | Source |
|---------------|-----------|--------|
| Target history | Lagged GW level at 1, 7, 14, 30 days | OVF monitoring well |
| Precipitation | Daily P, API at 7/14/30/60/90 days | ERA5 or OVF met stations |
| Temperature | Daily T_mean, T_max | ERA5 or OVF met stations |
| Evapotranspiration | Daily ET_0, cumulative P - ET_0 at 30/60/90 days | Computed from ERA5 |
| River influence | Nearest river stage at lags 1/7/14/30 days | OVF gauge stations |

| Trend | Year, linear trend | Computed |

### 23.3.3 Model Comparison: LSTM vs. Gradient Boosting vs. Simple Baselines

For groundwater level forecasting, the choice of ML algorithm is less clear-cut than for streamflow prediction. The slow dynamics and strong seasonality of groundwater levels mean that simple baselines -- persistence (tomorrow's level equals today's level) and seasonal climatology (this month's level equals the historical average for this month) -- already perform reasonably well. Any ML model must demonstrate clear improvement over these baselines to justify its additional complexity.

### 23.3.4 Global Perspective: Groundwater ML Beyond Hungary

The approaches described here for Hungary are directly applicable in any region with monitoring well data. The Global Groundwater Monitoring Network (GGMN), coordinated by IGRAC (International Groundwater Resources Assessment Centre), provides access to groundwater level time series from participating countries worldwide. In regions where monitoring density is low -- much of Sub-Saharan Africa, Central Asia, and Southeast Asia -- satellite-derived groundwater storage anomalies from the GRACE and GRACE-FO missions (see Chapter 3) provide an alternative target variable for ML models, albeit at a spatial resolution of approximately 300 km that limits their utility for local forecasting.

In data-rich regions, ML-based groundwater forecasting is becoming operational. The Netherlands' national groundwater monitoring system (Landelijk Meetnet Grondwaterkwaliteit / Grondwaterstanden) uses ensemble ML models for seasonal drought forecasting. In California, the Sustainable Groundwater Management Act (SGMA) has driven the development of ML forecasting tools for the state's 515 medium- and high-priority groundwater basins. In India, the Central Ground Water Board's network of over 20,000 monitoring wells has been used to train national-scale ML models for monsoon-season groundwater level prediction, with practical applications for rural drinking water security.

## 23.4 Water Quality Prediction from Sensor Data

### 23.4.1 The Monitoring Challenge

Water quality monitoring presents a different ML challenge from hydrologic forecasting. The target variables -- turbidity, dissolved oxygen, nutrient concentrations (nitrate, phosphate, ammonium), chlorophyll-a, pH -- are measured by in-situ sensors that vary enormously in reliability, cost, and maintenance requirements. Optical sensors for turbidity and chlorophyll-a are relatively robust and can operate unattended for weeks; nutrient analyzers require regular reagent replacement and calibration; and many parameters of regulatory interest (heavy metals, organic micropollutants, pathogens) cannot be measured continuously at all but require laboratory analysis of grab samples collected at intervals of days to months.

This creates two distinct ML opportunities. First, "virtual sensor" models that predict hard-to-measure parameters from easy-to-measure ones: can we predict ammonium concentration from a combination of turbidity, conductivity, temperature, and discharge that are measured continuously? Second, spatial inference models that predict water quality at unmonitored locations from a network of monitored stations, satellite observations, and catchment characteristics.

### 23.4.2 Virtual Sensors: Predicting Nutrients from Surrogate Measurements

The virtual sensor concept relies on the physical and biogeochemical correlations between water quality parameters. Turbidity, for example, is correlated with total suspended sediment concentration, which in turn is correlated with particulate phosphorus. Conductivity reflects the total dissolved ion load, which includes nitrate and other nutrients. Discharge influences dilution, and temperature controls biological reaction rates. A model that learns these correlations from a period of concurrent continuous and laboratory measurements can then generate continuous nutrient time series from the continuously measured surrogates alone.

On the Danube, the Joint Danube Survey and the ICPDR (International Commission for the Protection of the Danube River) water quality monitoring network provide extensive concurrent measurements at key stations including Budapest, Mohacs (southern Hungary), and dozens of stations upstream and downstream. The Budapest monitoring station, located below the Margaret Island (Margitsziget), measures turbidity, conductivity, temperature, pH, dissolved oxygen, and water level continuously (every 15 minutes), while grab samples are analyzed for nutrients (nitrate-N, ammonium-N, total phosphorus, orthophosphate) approximately weekly.

A random forest model trained on three years of concurrent data (2018--2020) to predict weekly ammonium concentration from the continuous surrogates achieves an $R^2$ of 0.73 and an RMSE of 0.041 mg/L on the 2021--2022 test period. When the same model is used to generate daily ammonium estimates from the continuous sensor data, it reveals pollution dynamics that the weekly grab sampling misses entirely -- short-lived concentration spikes during storm events and wastewater overflow episodes that last less than 48 hours.

|-----------------|-------|------|------|------------------------|
| Ammonium-N | 0.73 | 0.041 | mg/L | Turbidity, conductivity, temperature, discharge |

The model's accuracy for nitrate-N is notably higher than for ammonium-N, reflecting the more consistent relationship between nitrate and conductivity (nitrate is a major dissolved ion), compared to the more episodic and source-dependent behavior of ammonium (which spikes during combined sewer overflows and agricultural runoff events). The orthophosphate prediction is the weakest, because dissolved phosphorus concentrations in the Danube are low and variable, and the surrogate parameters capture only a fraction of the variance.

### 23.4.3 Spatial Water Quality Prediction

The second ML opportunity -- spatial inference -- addresses the reality that water quality monitoring networks are sparse. In Hungary, the OVF network includes approximately 140 surface water quality monitoring stations, but many of the country's smaller streams and channels are unmonitored. European regulations (the Water Framework Directive, WFD) require assessment of ecological and chemical status for all water bodies, including those without monitoring stations.

ML models can fill these spatial gaps by learning the relationship between catchment characteristics (land use, soil type, population density, wastewater treatment capacity, livestock density, fertilizer application rates) and water quality at monitored stations, then predicting water quality at unmonitored locations based on their catchment characteristics. This is essentially a spatial interpolation problem where the "interpolation" is done in feature space rather than geographic space.

The approach works best for parameters that are strongly controlled by catchment characteristics -- diffuse-source nutrients like nitrate (driven by agricultural land use and fertilizer application) -- and poorly for parameters controlled by point sources like industrial pollutants or pathogens. A gradient-boosted model trained on 100 Hungarian monitoring stations to predict mean annual nitrate-N from catchment descriptors (percentage agricultural land, mean soil organic carbon, livestock density, mean annual precipitation, baseflow index) achieves cross-validated $R^2$ of approximately 0.65. This is not sufficient for regulatory compliance monitoring, but it is useful for screening: identifying unmonitored streams that are likely to exceed the 50 mg/L nitrate limit and should be prioritized for direct monitoring.

### 23.4.4 Water Quality and Satellite Remote Sensing

For lakes and large rivers, satellite remote sensing adds another dimension to water quality prediction. Sentinel-2's MultiSpectral Instrument (MSI) provides 10 m resolution imagery in bands sensitive to chlorophyll-a (indicative of algal biomass), colored dissolved organic matter (CDOM), and suspended sediment. Machine learning models trained on concurrent satellite and in-situ measurements can produce spatially continuous water quality maps at the satellite's revisit frequency (5 days for Sentinel-2A+B combined).

In Hungary, this is directly relevant for Lake Balaton, where eutrophication management has been a priority since the 1980s. A random forest model trained on 2017--2023 concurrent Sentinel-2 reflectance and in-situ chlorophyll-a measurements from the four Balaton sub-basins (Keszthely, Szigliget, Szemes, Siofok) achieves RMSE of approximately 3.5 ug/L and captures the well-known west-to-east trophic gradient (the shallow western Keszthely basin is consistently more eutrophic than the eastern Siofok basin). The satellite-derived maps reveal spatial patterns -- blooms initiating along specific shoreline segments, interaction between the Zala River inflow and the Keszthely basin -- that point-based monitoring at fixed stations cannot capture.

For the Danube, satellite-based turbidity estimation using Sentinel-2 is operationally useful for sediment transport monitoring and for identifying tributary contributions to the main channel's sediment load. The ML approach consistently outperforms simple band-ratio algorithms (e.g., single-band NIR or red-edge algorithms) because it can adapt to the variable water composition of a large river that receives inputs from Alpine, Carpathian, and lowland tributaries with very different optical properties.

## 23.5 Land Cover Classification for Hydrologic Modeling

### 23.5.1 Why Land Cover Matters for Hydrology

Every hydrologic model, whether physically based or data-driven, needs to know what covers the land surface. Land cover determines interception (how much rainfall is captured by vegetation before reaching the soil), infiltration (which depends on soil cover and root systems), roughness (which controls overland flow velocity and channel resistance), and evapotranspiration (which varies by orders of magnitude between bare soil, grassland, forest, and open water). In the curve number method (Chapter 8), land cover is one of two inputs -- alongside hydrologic soil group -- that determines the runoff coefficient. In distributed hydrologic models like HEC-HMS or MIKE SHE, every grid cell must be assigned land cover-dependent parameters for interception storage, leaf area index, root depth, and surface roughness.

For many applications, existing land cover products are sufficient. The ESA WorldCover product provides global 10 m resolution land cover from Sentinel-1 and Sentinel-2 data, updated annually. The Copernicus CORINE Land Cover (CLC) dataset, a cornerstone of European environmental monitoring, provides 25 m resolution land cover for all EU member states in a standardized 44-class nomenclature, updated every six years (most recently CLC2018, with CLC2024 in production). Hungary's national CORINE maps are produced by the Lechner Tudaskoezpont and are freely downloadable.

However, there are situations where existing products are inadequate and the hydrologist needs to create a custom land cover classification. These include: (1) when the required thematic resolution exceeds what existing products offer -- distinguishing irrigated from rainfed cropland, for example, or identifying specific crop types; (2) when the temporal resolution is insufficient -- CORINE updates every six years, but land cover can change dramatically within a single growing season due to crop rotation, urbanization, or natural disturbance; and (3) when the existing product has known errors in the study area, which is common in heterogeneous landscapes where the classification algorithm's training data was sparse. In all these cases, the hydrologist must perform their own classification, and machine learning provides the most effective tools.

### 23.5.2 Pixel-Based Classification with Random Forest

The workhorse of satellite image classification in hydrology is the random forest algorithm, a ensemble of decision trees that is fast, robust, and remarkably effective for multi-class classification of multi-spectral imagery. The typical workflow for creating a hydrologically relevant land cover map from Sentinel-2 proceeds through five steps.

Step one: acquire and preprocess the imagery. A cloud-free (or nearly so) Sentinel-2 Level-2A (surface reflectance) composite for the study area during the peak growing season provides the best class separability. For Hungary, July or August composites typically work best for distinguishing cropland types. Sentinel-2 provides 13 spectral bands ranging from visible (490 nm) to shortwave infrared (2190 nm) at resolutions of 10, 20, and 60 m. For classification, the 10 m bands (B2, B3, B4, B8) and the 20 m bands (B5, B6, B7, B8A, B11, B12) are resampled to a common 10 m grid.

Step two: compute spectral indices. In addition to the raw bands, derived indices significantly improve classification accuracy. The most useful for hydrologic applications are NDVI (distinguishes vegetated from non-vegetated surfaces), NDWI (Normalized Difference Water Index, distinguishes water bodies), NDBI (Normalized Difference Built-up Index, distinguishes urban areas), and the Bare Soil Index (BSI, distinguishes exposed soil from vegetation and water).

Step three: create training data. This is the most labor-intensive step and the one that most determines classification accuracy. Training polygons are digitized for each target class, using high-resolution imagery (Google Earth, national orthophotos), field knowledge, and existing CORINE data as reference. For a hydrologically relevant classification with 8--10 classes (water, urban, deciduous forest, coniferous forest, grassland, irrigated cropland, rainfed cropland, bare soil, wetland), approximately 100--200 training polygons per class are needed, distributed across the study area to capture intra-class variability.

Step four: train the random forest. The model is trained on the spectral values (bands plus indices) extracted from the training polygons. Typical hyperparameters are 200--500 trees, maximum tree depth of 15--20, and a minimum leaf size of 5--10 samples. The out-of-bag (OOB) error estimate, an internal cross-validation metric unique to random forests, provides an immediate accuracy assessment without a separate validation set.

Step five: classify the full image and validate. The trained model predicts a class label for every pixel in the study area. Validation uses an independent set of ground truth points (not used in training) to construct a confusion matrix and compute overall accuracy, kappa coefficient, and per-class user's and producer's accuracies.

A typical result for a Sentinel-2-based random forest classification of the Middle Tisza region (approximately 5,000 km2 centered on Szolnok) achieves overall accuracy of 85--90% with an 8-class scheme. The most commonly confused classes are rainfed cropland and grassland (spectrally similar during certain phenological stages) and irrigated and rainfed cropland (distinguishable only when irrigation creates a moisture difference visible in the shortwave infrared bands). Adding multi-temporal imagery -- spring and autumn scenes in addition to the summer scene -- improves the discrimination of cropland types and raises overall accuracy to 90--93%.

### 23.5.3 Deep Learning: U-Net for Semantic Segmentation

While random forest treats each pixel independently, deep learning approaches -- specifically convolutional neural networks (CNNs) -- classify pixels in the context of their spatial neighborhood. The U-Net architecture, originally developed for biomedical image segmentation but now widely adopted in remote sensing, processes image patches (e.g., 256 x 256 pixels) and produces a classification map of the same size. The architecture consists of an encoding path (successive convolutional and pooling layers that capture increasingly abstract spatial features) and a decoding path (upsampling layers that restore the spatial resolution), connected by skip connections that preserve fine-grained spatial detail.

The advantage of U-Net over random forest is most pronounced for classes defined by spatial patterns rather than spectral properties alone. Urban areas, for example, have characteristic textures (regular building footprints, road networks) that a pixel-based classifier cannot exploit. Similarly, wetlands often have a distinctive spatial structure -- a mosaic of water, emergent vegetation, and wet grassland -- that is more reliably identified by a CNN that sees the pattern than by a random forest that classifies each pixel in isolation.

For Hungarian applications, a U-Net trained on Sentinel-2 imagery for the entire Tisza catchment within Hungary achieves 3--5 percentage points higher overall accuracy than random forest, with the most significant improvements in the wetland and urban classes. The marginal improvement may seem modest, but for hydrologic modeling applications where wetland extent directly influences flood attenuation and baseflow generation, even a few percentage points of improved wetland classification accuracy can substantially affect model predictions.

The practical barrier to U-Net adoption is the requirement for large, densely labeled training datasets. Random forest can be trained on scattered point or polygon samples; U-Net requires fully labeled image patches where every pixel has a class label. Creating such training data manually is prohibitively expensive for large areas. The solution is to use existing land cover products (CORINE, WorldCover) as "noisy labels" for initial training, then refine the model using a smaller set of manually verified patches. This semi-supervised approach achieves accuracy close to fully supervised training while reducing the labeling effort by an order of magnitude.

## 23.6 Transfer Learning: From Data-Rich to Data-Poor Catchments

### 23.6.1 The Data Scarcity Problem

The applications described so far assume abundant training data: long streamflow records, dense flux tower networks, extensive water quality monitoring. But many of the world's catchments -- and some within Hungary -- lack this data richness. Small tributaries of the Tisza in the Northern Hungarian Mountains (Bukk, Matra) may have only a few years of sporadic discharge measurements. Newly established monitoring wells may have records too short for reliable model training. And in developing countries, where the need for hydrologic forecasting is often greatest, long-term monitoring networks may be sparse or non-existent.

Transfer learning addresses this problem by leveraging knowledge gained from data-rich catchments to improve predictions in data-poor ones. The concept is borrowed from computer vision, where models trained on millions of labeled images (ImageNet) are routinely fine-tuned for specialized tasks with much smaller datasets. In hydrology, the equivalent is a model trained on hundreds of well-gauged catchments that can be adapted to a new catchment with minimal local data.

### 23.6.2 How Transfer Learning Works in Hydrology

The Kratzert regional LSTM described in Section 23.1.4 is already a form of implicit transfer learning: by training on many catchments simultaneously with static attributes as inputs, the model learns a general rainfall-runoff relationship that can be applied to any catchment for which the attributes are known, even if that catchment contributed no training data. The performance at ungauged catchments (those excluded from training) is, unsurprisingly, worse than at gauged catchments -- but it is often better than traditional regionalization approaches that estimate conceptual model parameters from physical catchment characteristics using regression equations.

Explicit transfer learning goes further. The typical approach is:

1. **Pre-training**: Train an LSTM (or other model) on a large, diverse set of catchments from a data-rich region. This model learns general hydrologic behaviors: the relationship between rainfall intensity and runoff response, the effect of temperature on snowmelt, the seasonal dynamics of evapotranspiration.

2. **Fine-tuning**: Take the pre-trained model and continue training on the limited data available for the target catchment. The fine-tuning process adjusts the model's weights to capture the specific characteristics of the target catchment while retaining the general hydrologic knowledge from pre-training.

3. **Regularization**: To prevent the fine-tuning from overfitting to the small local dataset, the learning rate is typically reduced (often by a factor of 10--100 compared to pre-training), and only certain layers are allowed to update (typically the output layers, while the input and hidden layers remain frozen or are updated very slowly).

The effectiveness of transfer learning depends critically on the hydrologic similarity between the source and target regions. A model pre-trained on Alpine catchments in Austria transfers well to Carpathian catchments in Romania and northern Hungary (similar climate, geology, and elevation range) but poorly to arid catchments in the Middle East (fundamentally different rainfall-runoff processes). This motivates the use of hydrologic similarity metrics -- shared climate types (Koppen classification), similar aridity indices, comparable catchment sizes and slopes -- when selecting source catchments for transfer.

### 23.6.3 Practical Implementation

Implementing transfer learning for a data-poor catchment requires access to a pre-trained model and a set of static catchment attributes for the target catchment. The CAMELS and LamaH-CE datasets provide both the training data and the attribute definitions needed for the source domain. For a Hungarian catchment not included in LamaH-CE, the hydrologist must compute the required attributes from available data:

- **Topographic attributes** (area, mean elevation, mean slope): from DEM analysis, as described in Chapters 5--7
- **Soil attributes** (depth, clay fraction, organic carbon): from the European Soil Database (ESDAC) or the Hungarian AGROTOPO digital soil map
- **Land cover fractions**: from CORINE or WorldCover, as described in Section 23.5
- **Climate attributes** (mean annual precipitation, aridity index, fraction of snow): from ERA5 reanalysis or OVF station data
- **Geological attributes** (dominant lithology, permeability class): from the 1:100,000 geological map of Hungary (MFGI)

Once the attributes are computed, they are concatenated with the time-varying meteorological inputs and fed into the pre-trained LSTM. If local streamflow data is available (even a few months), the fine-tuning step is performed. The resulting model provides discharge predictions that, while not as accurate as a locally calibrated model with decades of data, are far better than what would otherwise be available for an ungauged catchment.

## 23.7 Practical Considerations for ML in Hydrology

### 23.7.1 Data Splitting: Why Random Splits Are Wrong

The single most common methodological error in applied ML for hydrology is the use of random train/test splits. In a standard ML textbook, the data is randomly divided into training, validation, and test sets -- typically 70/15/15 or 80/10/10. This works when the data points are independent and identically distributed (i.i.d.). Hydrologic time series are emphatically not i.i.d.: discharge today is correlated with discharge yesterday and last week. A random split that places Monday's data in the training set and Tuesday's in the test set allows the model to "cheat" by exploiting this temporal autocorrelation, producing test-set performance metrics that dramatically overestimate the model's true predictive skill.

The correct approach for time series data in hydrology is a temporal split: all training data comes from one continuous time period, all validation data from a later period, and all test data from the most recent period. This ensures that the model is always evaluated on data it has never seen and that falls entirely outside its training window. A common split for a 20-year record might be:

- Training: years 1--14 (70%)
- Validation: years 15--17 (15%)
- Test: years 18--20 (15%)

For studies that claim generalization across catchments (as in the CAMELS experiments), an additional spatial split is needed: some catchments are held out entirely from training and used only for testing. The combination of temporal and spatial holdout provides the most conservative -- and most honest -- estimate of model performance.

For groundwater level forecasting, where the autocorrelation timescale is weeks to months rather than days, even a temporal split can be insufficiently conservative if the gap between training and test periods is too short. A minimum gap of 30--90 days between the end of the training period and the start of the test period (a "buffer zone") ensures that the test performance is not inflated by residual correlation.

### 23.7.2 Overfitting: The Silent Killer

Overfitting occurs when a model learns the noise in the training data rather than the underlying signal, producing excellent training-set performance but poor generalization to new data. In hydrology, overfitting is particularly dangerous because hydrologic datasets are often short (a few decades at most), contain non-stationarities (climate change, land use change, infrastructure development), and include rare extreme events that are by definition underrepresented in the training data.

The symptoms of overfitting are:

- A large gap between training and validation/test performance (e.g., training NSE 0.98, test NSE 0.72)
- The model captures noise patterns in the training set that do not recur in the test set
- Performance degrades as the forecast horizon increases
- The model produces implausible predictions for input values outside the training range (e.g., negative discharge, impossibly high ET)

The remedies, in order of importance:

**More data.** The most effective regularizer is a larger training set. For hydrologic time series, this means using longer records (if available), incorporating data from multiple stations, or using data augmentation techniques (adding noise, using bootstrap resampling of hydrologic events).

**Simpler models.** A model with fewer parameters is less prone to overfitting. Before reaching for a 3-layer, 512-unit LSTM, try a single-layer, 64-unit LSTM. Before trying deep learning, try XGBoost. Before trying XGBoost, try linear regression. If the simpler model performs nearly as well on the test set, prefer it.

**Regularization techniques.** For neural networks: dropout (randomly zeroing a fraction of neuron activations during training), weight decay (L2 regularization penalizing large weights), and early stopping (terminating training when validation loss stops decreasing). For tree-based models: limiting tree depth, setting minimum leaf sizes, and using subsampling. These techniques are standard in any ML framework and should always be employed.

**Cross-validation.** Instead of a single temporal split, use multiple splits: train on years 1--14 / test on 15--20, then train on years 1--10 + 15--20 / test on 11--14, and so on. This rolling or expanding window cross-validation provides multiple independent estimates of test performance and reveals whether the model's accuracy is consistent across different time periods or is sensitive to the specific events in the test set.

### 23.7.3 Interpretability: Opening the Black Box

The "black box" criticism is the most common objection to ML in operational hydrology. A physically based model like HEC-HMS or MODFLOW can explain its predictions: "Discharge increased because rainfall exceeded the infiltration capacity and generated Hortonian overland flow." An LSTM's explanation is, at best, "The weights in the network produced this output." For operational decision-making -- especially in flood forecasting, where incorrect predictions can cost lives -- this lack of interpretability is a legitimate concern.

Several techniques help open the black box, each providing a different type of insight:

**Feature importance** (for tree-based models). Random forest and XGBoost provide importance scores for each input feature, indicating how much each feature contributes to the model's predictions. As shown in Section 23.2.4, these importances often align with physical expectations, providing reassurance that the model has learned meaningful relationships.

**SHAP (SHapley Additive exPlanations).** SHAP values, based on cooperative game theory, provide instance-level explanations: for each individual prediction, which features pushed the prediction higher or lower, and by how much? For a flood prediction, SHAP can reveal that the model's high-discharge prediction is driven primarily by heavy upstream rainfall two days ago and elevated antecedent soil moisture, which is a physically sensible explanation. SHAP is available as a Python library and works with any model type (tree-based, neural network, or other).

**Attention mechanisms.** For LSTM models, adding an attention layer reveals which time steps in the input sequence the model "pays attention to" when making its prediction. In flood forecasting, the attention weights typically peak at time steps corresponding to the flood wave travel time from upstream stations -- exactly what a hydrologist would expect. This provides both interpretability and a diagnostic: if the attention peaks at unexpected time steps, the model may have learned a spurious correlation.

**Partial dependence plots.** These show how the model's prediction changes as a single input feature varies, with all other features held constant. For an ET model, a partial dependence plot of ET versus NDVI should show a monotonically increasing relationship (more vegetation, more transpiration), and a plot of ET versus LST should show a decreasing relationship (cooler surfaces have higher ET). If these relationships are not observed, the model has learned something unexpected that warrants investigation.

The key message is that interpretability is not an afterthought but a requirement. A model that makes accurate predictions for physically interpretable reasons is more trustworthy -- and more likely to generalize to new conditions -- than one that achieves the same accuracy through opaque and potentially spurious correlations.

### 23.7.4 When ML Fails: Knowing the Limits

Machine learning models in hydrology fail predictably in several scenarios that the practitioner must be aware of:

**Extrapolation beyond training data.** ML models interpolate well but extrapolate poorly. If the training set contains no discharge values above 2,000 m3/s, the model's prediction for a 3,000 m3/s event is unreliable. This is the fundamental limitation for extreme event prediction: the events we most need to predict are, by definition, the ones least represented in the training data. Physically based models, while imperfect, can extrapolate beyond observed ranges because their equations represent processes that continue to operate at extreme values. A hybrid approach -- using ML for routine forecasting and switching to a physically based model when predicted values approach the edge of the training distribution -- is one pragmatic solution.

**Non-stationarity.** If the catchment has changed -- a dam was built, a forest was cleared, the climate shifted -- the relationships learned from historical data may no longer hold. ML models assume that the future resembles the past. When this assumption is violated, the model degrades silently. The physical model, by contrast, can accommodate known changes (e.g., adding a reservoir routing module after dam construction).

**Data quality issues.** ML models absorb the biases and errors in their training data. If the streamflow record contains rating curve errors during high flows (a common problem, as discussed in Chapter 16), the model will learn those errors. If the meteorological inputs have systematic biases (e.g., undercatch of precipitation by gauges during snow events), the model will compensate by learning a biased rainfall-runoff relationship that works only as long as the bias persists.

**Misleading correlations.** In a multivariate dataset, ML models can latch onto correlations that are statistically strong but physically meaningless. The classic example: a model might learn that holiday-season dates predict lower industrial water discharge -- not because holidays affect hydrology, but because industrial abstraction drops during holidays. If holidays shift or industrial patterns change, the model breaks. Feature selection guided by physical understanding, and post-hoc verification of learned relationships through SHAP or partial dependence plots, are essential safeguards.

## 23.8 Tools and Frameworks

### 23.8.1 Python as the Common Platform

The tools of machine learning in hydrology are, without exception, Python-based. This is not because Python is the fastest or most elegant language, but because the ML research community standardized on Python in the 2010s and the resulting ecosystem of libraries -- for data manipulation, model training, visualization, and deployment -- is unmatched. The hydrologist who invests in learning Python (even at a basic level) gains access to the entire ML toolkit; the one who insists on working exclusively in MATLAB, R, or Excel will find the available tools increasingly limited.

The essential Python libraries for hydrologic ML work form a stack:

**Data handling**: NumPy (numerical arrays), pandas (tabular time series data), xarray (multi-dimensional gridded data such as satellite images and reanalysis fields).

**Visualization**: Matplotlib (general-purpose plotting), seaborn (statistical visualization), cartopy or folium (geospatial visualization).

**Classical ML**: scikit-learn (the standard library for random forest, gradient boosting, support vector machines, clustering, and all associated utilities: train/test splitting, cross-validation, hyperparameter tuning, pipeline construction). For most hydrologic ML applications that do not require deep learning, scikit-learn is sufficient.

**Gradient boosting**: XGBoost and LightGBM, which provide faster and more accurate gradient-boosted tree models than scikit-learn's built-in implementation. These are the go-to tools for tabular prediction problems (groundwater levels, water quality, spatial prediction).

**Deep learning**: TensorFlow/Keras and PyTorch. TensorFlow, developed by Google, was the dominant framework until approximately 2020; PyTorch, developed by Meta, has since become the preferred choice in the research community due to its more intuitive programming model (define-by-run versus TensorFlow's original define-and-run approach). For the hydrologist, the practical difference is small: both can implement LSTMs, CNNs (for land cover classification), and encoder-decoder architectures with comparable ease. The NeuralHydrology library, developed by Kratzert's group at JKU Linz, provides a PyTorch-based framework specifically designed for hydrologic LSTM experiments, with built-in support for CAMELS and LamaH-CE data formats.

**Geospatial ML**: rasterio and GDAL (reading/writing satellite imagery), geopandas (vector geospatial data), and rasterstats (extracting raster statistics for vector features).

### 23.8.2 scikit-learn for the Hydrologist

For the hydrologist new to ML, scikit-learn is the right starting point. Its API is consistent, well-documented, and designed around a simple pattern: create a model object, call `.fit()` with training data, call `.predict()` with new data. A complete random forest classification of a Sentinel-2 image, from data loading to accuracy assessment, can be implemented in fewer than 50 lines of Python.

The key scikit-learn tools for hydrologic applications include:

```
from sklearn.ensemble import RandomForestClassifier, RandomForestRegressor
from sklearn.ensemble import GradientBoostingRegressor
from sklearn.model_selection import TimeSeriesSplit
from sklearn.preprocessing import StandardScaler
from sklearn.metrics import mean_squared_error, r2_score
from sklearn.pipeline import Pipeline
```

The `TimeSeriesSplit` function is particularly important: it implements temporal cross-validation that respects the ordering of hydrologic data, avoiding the random-split trap described in Section 23.7.1. Each fold uses a progressively larger training set and a fixed-size validation set that always follows the training period in time.

A minimal workflow for groundwater level prediction with scikit-learn:

```python
import pandas as pd
from sklearn.ensemble import GradientBoostingRegressor
from sklearn.model_selection import TimeSeriesSplit
from sklearn.preprocessing import StandardScaler
from sklearn.metrics import mean_squared_error, r2_score
import numpy as np

# Load data (columns: date, gw_level, precip, temp, river_stage, ...)
df = pd.read_csv('well_data.csv', parse_dates=['date'], index_col='date')

# Feature engineering
df['API_30'] = df['precip'].rolling(30).sum()
df['API_90'] = df['precip'].rolling(90).sum()
df['doy_sin'] = np.sin(2 * np.pi * df.index.dayofyear / 365)
df['doy_cos'] = np.cos(2 * np.pi * df.index.dayofyear / 365)
df['gw_lag30'] = df['gw_level'].shift(30)

# Define features and target (30-day-ahead prediction)
feature_cols = ['precip', 'temp', 'river_stage', 'API_30',
                'API_90', 'doy_sin', 'doy_cos', 'gw_lag30']
df['target'] = df['gw_level'].shift(-30)  # 30 days into the future
df = df.dropna()

X = df[feature_cols]
y = df['target']

# Temporal split (last 3 years for testing)
train_end = '2019-12-31'
X_train, y_train = X[:train_end], y[:train_end]
X_test, y_test = X['2020-01-01':], y['2020-01-01':]

# Scale features
scaler = StandardScaler()
X_train_s = scaler.fit_transform(X_train)
X_test_s = scaler.transform(X_test)

# Train model
model = GradientBoostingRegressor(
    n_estimators=300, max_depth=5, learning_rate=0.05,
    subsample=0.8, random_state=42
)
model.fit(X_train_s, y_train)

# Evaluate
y_pred = model.predict(X_test_s)
print(f"R2: {r2_score(y_test, y_pred):.3f}")
print(f"RMSE: {np.sqrt(mean_squared_error(y_test, y_pred)):.2f} cm")
```

This example is deliberately minimal -- a production system would include hyperparameter tuning, cross-validation, feature selection, and uncertainty estimation -- but it illustrates the essential workflow: load data, engineer features, split temporally, scale, train, evaluate. The entire script runs in seconds on a modern laptop.

### 23.8.3 TensorFlow and PyTorch: When You Need Deep Learning

Deep learning is warranted when the problem involves sequential data with long-range dependencies (LSTMs for streamflow) or spatial pattern recognition (CNNs for land cover). Both TensorFlow and PyTorch can implement these architectures; the choice between them is largely a matter of personal preference and institutional support.

For the hydrologist who wants to train an LSTM for discharge forecasting, the NeuralHydrology library (https://neuralhydrology.readthedocs.io/) provides the most direct path. It handles data loading from CAMELS and LamaH-CE formats, implements multiple LSTM variants (standard, encoder-decoder, multi-timescale), supports static attribute inputs, and includes evaluation metrics specific to hydrology (NSE, KGE, alpha-NSE decomposition). Training a regional LSTM on the CAMELS dataset with NeuralHydrology requires writing a configuration file (specifying the data path, model architecture, training parameters, and evaluation metrics) and running a single command. The library abstracts away the PyTorch boilerplate, allowing the hydrologist to focus on the hydrologic decisions (which inputs, which catchments, which training period) rather than the software engineering.

For land cover classification with U-Net, the segmentation_models_pytorch library provides pre-implemented U-Net and similar architectures with pre-trained encoders (ResNet, EfficientNet) that can be fine-tuned on satellite imagery. The pre-trained encoder approach is itself a form of transfer learning: the encoder, originally trained on millions of natural images, already knows how to extract useful spatial features (edges, textures, shapes) that are relevant for satellite image classification.

### 23.8.4 Google Earth Engine: ML at Planetary Scale

Google Earth Engine (GEE) -- introduced in Chapter 3 and used throughout this book for data access -- also provides built-in ML capabilities that are particularly useful for two tasks: land cover classification and spatial prediction at continental to global scales.

GEE's ML API supports random forest, gradient-boosted trees, support vector machines, and naive Bayes classifiers, all running on Google's cloud infrastructure against the entire Sentinel-2, Landsat, and MODIS archive. The advantage is that the data never leaves the cloud: the analyst writes a JavaScript or Python script that defines the training data, selects the image bands, trains the classifier, and generates the classified map, all without downloading a single satellite scene. For a land cover classification of the entire Tisza catchment (approximately 157,000 km2 across five countries) from Sentinel-2, GEE can produce the classified map in minutes, whereas a desktop-based workflow would require downloading terabytes of imagery and processing it on local hardware.

A simplified GEE workflow for land cover classification:

```javascript
// Define study area
var tisza = ee.FeatureCollection('users/example/tisza_basin');

// Create cloud-free Sentinel-2 composite for summer 2023
var s2 = ee.ImageCollection('COPERNICUS/S2_SR_HARMONIZED')
  .filterBounds(tisza)
  .filterDate('2023-06-01', '2023-08-31')
  .filter(ee.Filter.lt('CLOUDY_PIXEL_PERCENTAGE', 20))
  .median()
  .clip(tisza);

// Compute spectral indices
var ndvi = s2.normalizedDifference(['B8', 'B4']).rename('NDVI');
var ndwi = s2.normalizedDifference(['B3', 'B8']).rename('NDWI');
var image = s2.select(['B2','B3','B4','B8','B11','B12'])
  .addBands(ndvi).addBands(ndwi);

// Load training data (point features with 'class' property)
var training = ee.FeatureCollection('users/example/training_points');

// Sample the image at training points
var trainData = image.sampleRegions({
  collection: training,
  properties: ['class'],
  scale: 10
});

// Train Random Forest classifier
var classifier = ee.Classifier.smileRandomForest(200)
  .train(trainData, 'class');

// Classify the image
var classified = image.classify(classifier);

// Export to Google Drive
Export.image.toDrive({
  image: classified,
  description: 'tisza_landcover_2023',
  region: tisza,
  scale: 10,
  maxPixels: 1e12
});
```

GEE also supports the export of training data for external ML: the analyst can sample satellite observations at gauge station locations, export the resulting table, and train more sophisticated models (XGBoost, LSTM) locally or on a cloud computing platform. This hybrid approach -- using GEE for data access and feature extraction, and Python for model training -- combines the best of both worlds.

## 23.9 Bringing It Together: A Decision Framework

The proliferation of ML methods, architectures, and tools can be overwhelming for the hydrologist encountering this field for the first time. The following decision framework provides a starting point for selecting the right approach for a given problem.

**What is the prediction task?**

If the task is time series forecasting (discharge, groundwater level, water quality at a single station), start with gradient-boosted trees (XGBoost) with engineered temporal features. If the temporal dynamics are complex (long memory, multiple timescales), try an LSTM. Use scikit-learn or XGBoost for the first approach, PyTorch/NeuralHydrology for the second.

If the task is spatial prediction (ET maps, water quality across a catchment, land cover classification), start with random forest in scikit-learn or GEE. If spatial context matters (texture, neighborhood patterns), try U-Net. Use GEE for large areas, local Python for small areas or when interpretability is needed.

If the task involves a data-poor catchment, use transfer learning: pre-train on a large-sample dataset (CAMELS, LamaH-CE, Caravan), then fine-tune on local data.

**How much data is available?**

Fewer than 5 years of daily data: use simpler models (linear regression, random forest with few features). Avoid deep learning, which requires more data to avoid overfitting.

5--20 years: the sweet spot for most ML approaches. Gradient boosting and LSTM both work well.

More than 20 years, multiple catchments: ideal for regional LSTM or transfer learning experiments.

**Is interpretability required?**

For operational forecasting where decisions affect public safety: use tree-based models with SHAP explanations, or LSTM with attention mechanisms. Always provide uncertainty estimates.

For research or screening purposes: any model that achieves good test-set performance is acceptable, but always verify learned relationships against physical expectations.

**What computational resources are available?**

A standard laptop: scikit-learn, XGBoost (training in seconds to minutes). Small LSTMs (training in minutes to hours).

A workstation with GPU: large LSTMs, U-Net for satellite image classification. Training in hours.

Cloud resources (Google Colab, institutional HPC): regional LSTMs on large-sample datasets. GEE for planetary-scale analyses.

The overarching principle is to start simple and add complexity only when justified by improved performance on a properly constructed test set. The most common mistake in hydrologic ML is not using too simple a model -- it is using too complex a model on too little data with an overly optimistic evaluation scheme.
