# Chapter 6: Calculating with Maps

Every chapter so far has treated raster grids as things to look at -- terrain surfaces to visualize, elevation values to query, datasets to organize and store. This chapter changes perspective. Here, the raster grid becomes something to compute with. The framework that makes this possible is called **map algebra**, and it is one of the most powerful ideas in spatial analysis: the notion that you can write an equation, apply it to an entire landscape at once, and get a new map as the result.

The idea is deceptively simple. If you have a map of how much rain fell and a map of how much water the soil absorbed, you can subtract one from the other and get a map of how much water is left on the surface -- the runoff. That single subtraction, applied simultaneously to every cell in a grid covering an entire watershed, is map algebra in its most elemental form. But the implications are profound. The same principle lets you implement the SCS Curve Number method across a 10,000-square-kilometer basin, combine satellite-derived soil moisture with weather radar precipitation to estimate flood risk in real time, or stack a dozen climate model outputs and compute their ensemble mean. Map algebra is the computational engine that turns static spatial data into dynamic spatial analysis.

This chapter begins with the conceptual framework -- what map algebra is and where the idea came from -- then moves to the practical tools for performing raster calculations in modern GIS software. The heart of the chapter is a detailed treatment of what happens when your input grids do not match: different cell sizes, different extents, different alignments. This is where most beginners make mistakes, and where most textbooks are too brief. We will work through the entire reconciliation process with explicit numerical examples, showing exactly which cell values end up in the output and why. We then cover the analysis environment settings that control this reconciliation, the different resampling methods and when to use each one, the handling of NODATA values, and a complete practical workflow for performing raster calculations correctly.

The concepts here are foundational for everything that follows. Chapter 7 uses map algebra to compute slope and aspect from elevation grids. Chapter 8 applies it to spatial interpolation of rainfall. Chapters 9 through 12 build entire hydrological workflows on raster calculations. If you understand this chapter thoroughly -- especially the cell size reconciliation material -- you will avoid a class of subtle errors that plague even experienced GIS analysts.

## 6.1 The Concept of Map Algebra

### 6.1.1 Origins and Principles

Map algebra was formalized by Dana Tomlin in the early 1980s as part of his doctoral work at Yale University, later published in his influential book *Geographic Information Systems and Cartographic Modeling* (1990). Tomlin recognized that raster grids could be treated as variables in algebraic expressions, just as numbers are treated in ordinary arithmetic. If a grid cell contains a number, and every cell in the grid contains a number, then the entire grid can participate in mathematical operations -- addition, subtraction, multiplication, division, exponentiation, logical comparisons, and more.

The fundamental principle is cell-by-cell correspondence. When you add two grids, the value in row 1, column 1 of the first grid is added to the value in row 1, column 1 of the second grid, and the result is stored in row 1, column 1 of the output grid. The same operation is performed independently at every cell position. There is no interaction between neighboring cells in a basic map algebra operation -- each cell's output depends only on the input values at that same cell location. This property makes map algebra operations embarrassingly parallel: they can be distributed across as many processor cores as you have, because no cell needs to know what is happening at any other cell.

Tomlin classified map algebra operations into three categories based on the spatial scope of the computation. **Local operations** compute each output cell from the corresponding cell(s) in one or more input grids -- this is the cell-by-cell arithmetic described above, and it is the focus of this chapter. **Focal operations** (also called neighborhood operations) compute each output cell from a window of surrounding cells in the input grid -- slope and aspect calculations (Chapter 7) are classic examples. **Zonal operations** compute a single summary value for each group of cells sharing a common zone identifier -- for example, computing the average elevation within each watershed polygon. **Global operations** compute each output cell using information from every cell in the input grid -- distance and cost-surface calculations are examples.

This classification matters because it determines the computational cost and the conceptual complexity of the operation. Local operations are the simplest and fastest. Focal operations require a moving window and are moderately expensive. Zonal operations require grouping and aggregation. Global operations can require iterative algorithms that sweep across the entire grid multiple times. In this chapter, we focus on local operations -- the algebraic heart of map algebra -- because they are the most commonly used in hydrology and the foundation for understanding everything else.

### 6.1.2 Why Map Algebra Matters for Hydrology

Hydrology is inherently spatial. Rainfall varies across a watershed. Soil properties change from hilltop to valley. Land cover transitions from forest to agriculture to urban pavement. Slope steepens and flattens. Every physically-based hydrological model must account for this spatial variability, and map algebra provides the mechanism for doing so within a GIS framework.

Consider the water balance equation, the most fundamental relationship in hydrology:

$$P = ET + R + \Delta S$$

where $P$ is precipitation, $ET$ is evapotranspiration, $R$ is runoff, and $\Delta S$ is the change in storage. In a lumped model, each of these terms is a single number representing the entire watershed. In a distributed model, each term is a grid -- a raster surface where every cell has its own value. The water balance is then solved cell by cell using map algebra:

$$R_{i,j} = P_{i,j} - ET_{i,j} - \Delta S_{i,j}$$

where the subscripts $i,j$ refer to the row and column position in the grid. This equation, applied simultaneously to every cell, produces a spatially distributed runoff map. The power is obvious: instead of a single runoff number for the watershed, you get a map showing exactly where runoff is generated, how much, and how the spatial pattern relates to the underlying terrain, soils, and land cover.

This distributed approach is not merely an academic exercise. Modern operational flood forecasting systems, including the US National Water Model, the European Flood Awareness System (EFAS), and national forecasting services worldwide, rely fundamentally on grid-based calculations. Weather radar provides precipitation on a grid. Satellite remote sensing provides soil moisture on a grid. Numerical weather prediction models provide forecast rainfall on a grid. The hydrological model processes all of these grids through map algebra operations to produce distributed runoff estimates, which are then routed through the stream network to produce flood forecasts at points of interest. Without map algebra, none of this would be computationally tractable.

### 6.1.3 The Simplicity of the Idea

Before diving into the technical details of software tools and cell size reconciliation, it is worth pausing to appreciate the conceptual elegance of map algebra. The idea is this: if you can write a formula that works for one location, you can apply it to a million locations simultaneously by expressing the inputs and outputs as grids.

Consider the simplest possible example. You have a 3x3 grid representing precipitation (in millimeters) and a 3x3 grid representing infiltration (in millimeters):

**Precipitation Grid (mm)**

| | Col 1 | Col 2 | Col 3 |
|---|---|---|---|
| **Row 1** | 50 | 45 | 60 |
| **Row 2** | 55 | 40 | 50 |
| **Row 3** | 48 | 52 | 44 |

**Infiltration Grid (mm)**

| | Col 1 | Col 2 | Col 3 |
|---|---|---|---|
| **Row 1** | 30 | 28 | 35 |
| **Row 2** | 32 | 25 | 30 |
| **Row 3** | 29 | 31 | 27 |

The map algebra expression $R = P - I$ produces:

**Runoff Grid (mm)**

| | Col 1 | Col 2 | Col 3 |
|---|---|---|---|
| **Row 1** | 20 | 17 | 25 |
| **Row 2** | 23 | 15 | 20 |
| **Row 3** | 19 | 21 | 17 |

Each value in the runoff grid is simply the precipitation value at that cell minus the infiltration value at that cell. Cell (1,1): $50 - 30 = 20$. Cell (2,3): $50 - 30 = 20$. Cell (3,2): $52 - 31 = 21$. No cell interacts with its neighbors. The operation is purely local, purely algebraic, and purely parallel.

This example uses nine cells. A real application might use nine million cells covering an entire river basin at 30-meter resolution. The principle is identical. The only difference is that the computer does the subtraction nine million times instead of nine, and it does so in seconds.

## 6.2 The Raster Calculator

### 6.2.1 Tools for Map Algebra

Every modern GIS platform provides tools for performing map algebra operations. The specific interface varies, but the underlying concept is always the same: you write an algebraic expression referencing input raster layers, and the software evaluates that expression cell by cell to produce an output raster.

In **ArcGIS Pro**, map algebra operations are performed using the **Raster Calculator** tool, found within the Spatial Analyst toolbox under the Map Algebra toolset. The Raster Calculator provides a dialog where you select input rasters from those loaded in the current map, build expressions using standard mathematical operators, specify an output raster name and location, and execute the calculation. The expression syntax follows Python conventions, and input rasters are referenced by enclosing their layer names in quotation marks. A typical expression might look like:

```
"precipitation" - "infiltration"
```

or, for a more complex formula:

```
("precipitation" - 0.2 * "retention") ** 2 / ("precipitation" + 0.8 * "retention")
```

The second expression implements the SCS Curve Number method for runoff estimation -- a formula you will encounter in detail in later chapters. The point here is that the Raster Calculator can handle arbitrarily complex algebraic expressions, not just simple arithmetic.

In **QGIS**, the equivalent tool is the **Raster Calculator** accessible from the Raster menu. The interface is similar: you select input layers, build an expression, and specify the output. QGIS uses its own expression syntax, where layer names are followed by the band number in the format `layername@1` (for band 1). The same subtraction would be written as:

```
"precipitation@1" - "infiltration@1"
```

QGIS also provides raster calculator functionality through the Processing framework, where GRASS GIS tools like `r.mapcalc` and SAGA tools offer additional map algebra capabilities with more sophisticated expression languages.

For **programmatic workflows**, both platforms support map algebra through scripting. In ArcGIS Pro, the `arcpy.sa` module provides a Pythonic interface where rasters can be used directly in mathematical expressions after importing the Spatial Analyst module. In QGIS, PyQGIS and the GDAL/NumPy interface allow raster calculations in Python scripts. For cloud-scale processing, Google Earth Engine provides a complete map algebra system operating on its planetary-scale raster catalog, and the Python libraries `rasterio` and `xarray` combined with `rioxarray` enable map algebra operations on any raster data accessible from a Python environment.

**Table 6.1**: Map algebra tools across GIS platforms

| Platform | Tool | Expression Syntax | Typical Use Case |
|----------|------|-------------------|------------------|
| ArcGIS Pro | Raster Calculator (Spatial Analyst) | `"layer" + "layer"` | Desktop analysis, model building |
| ArcGIS Pro | arcpy.sa (Python) | `Raster("a") + Raster("b")` | Scripted workflows, automation |
| QGIS | Raster Calculator | `"layer@1" + "layer@1"` | Desktop analysis |
| QGIS | GRASS r.mapcalc | `A + B` | Complex expressions, conditional logic |
| GDAL + Python | rasterio / NumPy | `array_a + array_b` | Custom scripting, batch processing |
| Google Earth Engine | ee.Image arithmetic | `image1.add(image2)` | Cloud-scale, global analysis |

The choice of tool depends on the context. For a one-off calculation on a few layers, the interactive Raster Calculator is the fastest approach. For a workflow that will be repeated many times or applied to dozens of input datasets, scripting is far more efficient. For analyses involving petabytes of satellite imagery, cloud platforms like Google Earth Engine are the only practical option. Regardless of platform, the analyst must understand the raster data model (Chapter 5) -- particularly cell size, data type, and NODATA conventions -- to use these tools correctly.

### 6.2.2 Output Formats and Data Types

When the Raster Calculator produces an output, the format of that output depends on how you name the file. In ArcGIS Pro, if you specify no file extension, the output is written in ESRI Grid format (a proprietary format stored as a folder of binary files). If you specify `.tif`, the output is a GeoTIFF -- the most widely used and interoperable raster format. GeoTIFF is strongly recommended for most work because it is supported by every GIS platform, stores coordinate system information internally, supports compression, and can be read by programming libraries in every language.

In QGIS, the output format is selected from a dropdown menu in the Raster Calculator dialog. GeoTIFF is the default and recommended choice. Other supported formats include ERDAS Imagine (`.img`), NetCDF (`.nc`), and various others through GDAL.

The **data type** of the output is equally important. Raster datasets can store values as integers (8-bit, 16-bit, 32-bit) or floating-point numbers (32-bit or 64-bit). When you subtract one precipitation grid from one infiltration grid, the result almost certainly requires floating-point storage, even if the inputs were integers, because the subtraction may produce fractional values. Most raster calculator tools automatically select an appropriate output data type, but you should verify this -- an accidental truncation to integer can produce silently wrong results. For example, if the true runoff at a cell is 1.775 mm but the output is stored as a 16-bit integer, the stored value will be 1 or 2, not 1.775. Over millions of cells, this rounding error can accumulate to significant bias in total runoff estimates.

**Table 6.2**: Common raster data types and their appropriate uses

| Data Type | Value Range | Storage per Cell | Appropriate For |
|-----------|-------------|------------------|-----------------|
| 8-bit unsigned integer | 0 to 255 | 1 byte | Land cover classes, small categorical datasets |
| 16-bit signed integer | -32,768 to 32,767 | 2 bytes | Elevation in meters (integer DEMs), classification codes |
| 32-bit signed integer | -2.1 billion to 2.1 billion | 4 bytes | Flow accumulation counts, large integer data |
| 32-bit floating point | ~7 significant digits | 4 bytes | Precipitation, temperature, most continuous data |
| 64-bit floating point | ~15 significant digits | 8 bytes | High-precision calculations, coordinate values |

### 6.2.3 Building Complex Expressions

The Raster Calculator is not limited to simple two-layer arithmetic. You can build expressions of arbitrary complexity, combining multiple layers with mathematical operators, parentheses for grouping, mathematical functions (square root, logarithm, trigonometric functions), and conditional logic.

A particularly useful capability is the **conditional (Con) expression**, which implements if-then-else logic on a cell-by-cell basis. In ArcGIS Pro's map algebra syntax:

```python
Con("slope" > 15, 1, 0)
```

This expression examines each cell of the slope raster. Where the slope exceeds 15 degrees, the output cell receives a value of 1; everywhere else, it receives 0. The result is a binary mask identifying steep terrain. Such masks are fundamental building blocks in hydrological analysis -- you might create a mask of areas where slope exceeds a threshold, where soil permeability is below a threshold, and where land cover is impervious, then multiply these masks together to identify the highest-risk runoff source areas.

In QGIS, conditional logic in the Raster Calculator uses a slightly different syntax based on boolean expressions:

```
("slope@1" > 15) * 1
```

This exploits the fact that boolean comparisons produce 1 (true) or 0 (false), which can then be multiplied by any value to create classified outputs.

More complex conditional logic can be built by chaining expressions. For instance, to classify runoff potential into three categories based on both slope and soil type:

```python
Con("slope" > 20, 3, Con("slope" > 10, 2, 1)) * Con("soil_perm" < 5, 2, 1)
```

This kind of multi-criteria classification is the basis for simple suitability analysis and risk mapping, topics that appear repeatedly in water resources applications.

## 6.3 The Precipitation-Minus-Infiltration Paradigm

### 6.3.1 A Central Metaphor

The subtraction of an infiltration grid from a precipitation grid is not just a textbook example -- it is the conceptual core of how distributed hydrological models work. Nearly every runoff generation method can be expressed as some variant of "what falls from the sky minus what the ground absorbs equals what flows across the surface." The details of how you estimate the infiltration (or "loss") term vary enormously -- from simple empirical methods like the SCS Curve Number to physically-based models like Green-Ampt or Richards' equation -- but the spatial implementation always reduces to map algebra.

In the simplest case, you have a grid of precipitation $P$ and a grid of losses $L$ (encompassing infiltration, interception, depression storage, and evapotranspiration during the storm), and runoff $R$ is:

$$R = \max(P - L, 0)$$

The $\max$ function ensures that runoff cannot be negative -- if losses exceed precipitation at a cell, there is simply no runoff there. In raster calculator syntax (ArcGIS Pro):

```python
Con("precipitation" > "losses", "precipitation" - "losses", 0)
```

This expression evaluates the condition $P > L$ at every cell. Where true, it computes $P - L$. Where false (losses exceed precipitation), it returns 0. The result is a physically meaningful runoff grid.

### 6.3.2 From Concept to Practice

The precipitation-minus-infiltration paradigm illustrates a broader pattern in how GIS-based hydrological modeling works. The analyst assembles a collection of input grids, each representing a spatially variable physical quantity, and then combines them through algebraic expressions that encode physical relationships. The power of this approach is that the same formula is applied everywhere, but the inputs vary spatially, so the results capture the spatial heterogeneity of the landscape.

In a realistic flood estimation workflow, the inputs might include:

- A radar-derived precipitation grid from the national weather service, at perhaps 1 km resolution, representing total storm rainfall (Chapter 8 discusses how such grids are created from point gauge data and satellite observations).
- A soil hydrologic group grid derived from a national soil survey, at perhaps 30 m resolution, classifying soils by their infiltration capacity (Group A through D).
- A land cover grid from satellite classification, at perhaps 10 m resolution, identifying impervious surfaces, cropland, forest, and other categories.
- A curve number lookup table that assigns a runoff coefficient to each combination of soil group and land cover type.

The workflow proceeds in stages, each stage being a map algebra operation:

1. **Create the Curve Number grid**: Use a conditional expression or lookup table to assign CN values to each cell based on its soil group and land cover combination.
2. **Compute the retention parameter**: $S = \frac{25400}{CN} - 254$ (in millimeters), applied cell by cell.
3. **Compute the initial abstraction**: $I_a = 0.2 \times S$, applied cell by cell.
4. **Compute the runoff**: $R = \frac{(P - I_a)^2}{P - I_a + S}$ where $P > I_a$, otherwise $R = 0$.

Each of these steps is a raster calculator expression. The final runoff grid captures the combined effects of spatially varying rainfall, soil properties, and land cover -- something that would be impossible with a lumped model using a single CN value for the entire watershed.

The critical challenge in this workflow, however, is not the algebra. It is the fact that the input grids have different cell sizes (1 km for precipitation, 30 m for soils, 10 m for land cover), different extents (the radar grid might cover the entire country while the soil map covers only one region), and potentially different coordinate systems. Before any map algebra can be performed, these grids must be reconciled to a common geometry. That reconciliation process -- cell size harmonization through resampling -- is the subject of the next section, and it is where most of the practical difficulty lies.

## 6.4 Cell Size Reconciliation and Resampling

### 6.4.1 The Problem of Mismatched Cell Sizes

In an ideal world, every raster dataset you need for an analysis would have the same cell size, the same extent, and the same grid alignment. In the real world, this almost never happens. Precipitation grids from weather radar typically have cell sizes of 1 to 4 kilometers. Digital elevation models come at 5 m (Hungarian DDM-5), 10 m (Copernicus EEA-10), 25 m (EU-DEM), 30 m (SRTM, Copernicus GLO-30), or 90 m (SRTM 3 arc-second). Soil maps might be rasterized from vector data at 30 m or 250 m. Land cover grids range from 10 m (ESA WorldCover, Sentinel-2 derived) to 300 m (ESA CCI-LC) to 1 km (MODIS land cover). When you need to combine these datasets in a map algebra expression, the software must reconcile the differences before it can perform any cell-by-cell calculation.

This reconciliation is not a minor bookkeeping detail. The choices made during reconciliation -- which cell size to use for the output, how to resample grids to match, which extent to adopt -- fundamentally affect the results. Using the wrong settings can silently discard data, introduce spatial biases, or produce results at an inappropriate resolution. Understanding how reconciliation works, and how to control it, is one of the most important practical skills in raster-based spatial analysis.

### 6.4.2 A Worked Example: Setting Up the Problem

To make the reconciliation process concrete and traceable, we will work through a complete numerical example using small grids where every cell value can be tracked by hand. Consider two input grids:

**Precipitation grid** (150 m cell size, 2x2 grid):

```
ncols         2
nrows         2
xllcorner     0
yllcorner     0
cellsize      150
NODATA_value  -9999
4     6
3.1   4
```

This represents a small area (300 m x 300 m) where the northwest cell received 4 mm of rain, the northeast cell received 6 mm, the southwest cell received 3.1 mm, and the southeast cell received 4 mm.

**Infiltration grid** (100 m cell size, 3x3 grid):

```
ncols         3
nrows         3
xllcorner     0
yllcorner     0
cellsize      100
NODATA_value  -9999
2.2   3     4
1.8   2.5   3.5
1.5   2     2.7
```

This represents the same geographic area, but at finer resolution (nine cells instead of four), with infiltration values ranging from 1.5 to 4 mm.

The desired calculation is straightforward: $Runoff = Precipitation - Infiltration$. But the two grids have different cell sizes (150 m vs. 100 m) and different numbers of cells (4 vs. 9). The precipitation grid has larger cells that each span an area covered by more than one infiltration cell. Before the subtraction can proceed, the software must decide: at what resolution will the output be computed, and how will the grid with the non-matching cell size be adjusted?

### 6.4.3 Default Behavior: What Happens Automatically

When you enter the expression `"precip" - "infiltration"` in the Raster Calculator without changing any environment settings, the GIS software applies default rules to reconcile the inputs. In ArcGIS Pro (and similarly in QGIS, though the exact defaults may differ slightly), the default behavior is:

1. **The output cell size is set to the coarsest input** -- in this case, 150 m (the precipitation grid).
2. **The output extent is set to the intersection** of the input extents -- the area covered by both grids.
3. **Nearest-neighbor resampling** is applied to any input that does not match the target cell size.

The result of this automatic reconciliation is that the infiltration grid (100 m) is resampled to 150 m using the nearest-neighbor method. This means that for each cell center in the output 150 m grid, the software finds the nearest cell center in the infiltration grid and uses that value. The infiltration values at cell centers that do not coincide with output cell centers are simply ignored.

Let us trace the result. The output is a 2x2 grid at 150 m resolution. The cell centers of the output grid are at:

- Cell (1,1): center at (75, 225) -- the upper-left cell
- Cell (1,2): center at (225, 225) -- the upper-right cell
- Cell (2,1): center at (75, 75) -- the lower-left cell
- Cell (2,2): center at (225, 75) -- the lower-right cell

For each output cell center, we find the nearest cell center in the infiltration grid (which has centers at x = 50, 150, 250 and y = 50, 150, 250):

- Output (1,1) at (75, 225): nearest infiltration center is (50, 250), value = 2.2
- Output (1,2) at (225, 225): nearest infiltration center is (250, 250), value = 4
- Output (2,1) at (75, 75): nearest infiltration center is (50, 50), value = 1.5
- Output (2,2) at (225, 75): nearest infiltration center is (250, 50), value = 2.7

The subtraction then produces:

**Table 6.3**: Default raster calculation result (150 m output, nearest-neighbor resampling)

| Output Cell | Precipitation | Nearest Infiltration | Runoff (P - I) |
|-------------|--------------|---------------------|----------------|
| (1,1) | 4.0 | 2.2 | 1.8 |
| (1,2) | 6.0 | 4.0 | 2.0 |
| (2,1) | 3.1 | 1.5 | 1.6 |
| (2,2) | 4.0 | 2.7 | 1.3 |

The output grid is:

```
1.8   2.0
1.6   1.3
```

This result is numerically correct given the defaults, but is it physically appropriate? Five of the nine original infiltration values (3.0, 1.8, 2.5, 3.5, 2.0) were never used in the calculation. They were discarded by the nearest-neighbor resampling. If the infiltration grid represented expensive field measurements or carefully calibrated model output, throwing away more than half the data is a significant loss. Furthermore, the output resolution (150 m) is coarser than the best available data (100 m), meaning the result contains less spatial detail than it could.

### 6.4.4 Verifying the Results

A crucial habit in raster analysis is to verify that the software produced the results you expected. The **Explore** (or **Identify**) tool in ArcGIS Pro and QGIS allows you to click on any point in the map and see the values of all visible raster layers at that location. Set the tool to display values from "All visible layers" rather than just the topmost layer.

By clicking at various points in the output runoff grid, you can verify individual cell values. At the center of the upper-left cell, you should see: precipitation = 4.0, infiltration (after resampling) = 2.2, runoff = 1.8. If you click at a location within the same 150 m output cell but closer to a different 100 m infiltration cell, the infiltration layer will show a different value (because you are querying the original 100 m grid), but the runoff layer will still show 1.8 (because it was computed at 150 m resolution from the resampled value).

This discrepancy -- the infiltration layer shows one value but the runoff was computed using a different value -- is a telltale sign that resampling has occurred. It is not an error in the software; it is the expected behavior of nearest-neighbor resampling at the coarse resolution. But it is a clear signal that you may want to take more control over the process.

### 6.4.5 Why the Default May Not Be What You Want

The default behavior -- coarsest cell size, nearest-neighbor resampling -- is computationally efficient and safe in a conservative sense: it does not fabricate detail that the coarsest input cannot support. But there are good reasons to override it:

**Information loss.** If your finest-resolution input represents expensive or carefully derived data (high-resolution soil survey, LiDAR-derived infiltration capacity, detailed land cover classification), reducing everything to the coarsest resolution throws away information you paid for and that might matter for the analysis.

**Spatial averaging vs. point sampling.** Nearest-neighbor resampling selects a single value from the fine grid to represent each coarse cell. A more physically meaningful approach for continuous data might be to average all the fine cells that fall within each coarse cell, or to use bilinear interpolation to estimate the value at the coarse cell center based on surrounding fine cells. Nearest-neighbor resampling is a point-sampling operation, not a spatial averaging operation, and for continuous physical quantities like infiltration, spatial averaging is often more appropriate.

**Resolution mismatch in the wrong direction.** Sometimes the coarsest input is not the one that should dictate the output resolution. If you are computing runoff to drive a detailed urban drainage model at 5 m resolution, you need the output at 5 m even though the precipitation input is at 1 km. In that case, you want to resample the precipitation to 5 m (accepting that it will be spatially smooth) and keep the fine-resolution land cover and soil data intact.

These considerations lead us to the analysis environment settings, which provide the controls for overriding the defaults.

## 6.5 The Analysis Environment

### 6.5.1 What the Analysis Environment Controls

Every raster geoprocessing tool in ArcGIS Pro and QGIS operates within an **analysis environment** -- a set of parameters that control how input grids are reconciled and how the output is generated. The three most important parameters for map algebra are:

**Analysis extent** determines the geographic area over which the calculation is performed. By default, this is the intersection of all input extents -- the area where every input has data. You can override this to use the extent of a specific input, the union of all inputs (filling missing areas with NODATA), or a manually specified rectangle. For watershed analysis, you often set the extent to match a watershed boundary or a reference DEM.

**Analysis cell size** determines the output resolution. By default, this is the maximum (coarsest) cell size among the inputs. You can override this to use the minimum (finest) cell size, match a specific input layer, or specify an explicit size in map units. This is the single most impactful setting for controlling the quality of raster calculations.

**Snap raster** specifies a reference raster whose cell alignment the output must match. Even if two grids have the same cell size, their cells may not be aligned -- one grid's cell boundaries might be offset by half a cell width relative to the other. Setting a snap raster ensures that the output cells line up exactly with a reference grid, which is essential for maintaining consistency across a series of raster operations.

A fourth parameter, the **analysis mask**, restricts the calculation to a specific geographic region. Cells outside the mask receive NODATA in the output. The mask can be a raster (where non-zero, non-NODATA cells define the active area) or a feature class (where the polygon geometry defines the active area). Mask layers are commonly used to restrict analysis to a watershed boundary, excluding areas outside the drainage basin that are irrelevant to the hydrological question.

### 6.5.2 Setting the Analysis Environment in Practice

In **ArcGIS Pro**, the analysis environment is set in two ways. For an individual tool, click the **Environments** tab at the bottom of the tool dialog to set environment parameters for that specific execution. For a global default that applies to all tools, go to the **Analysis** tab on the ribbon and click **Environments** to open the project-wide settings. Tool-specific settings override project-wide settings.

The relevant settings for raster calculations are found under the **Raster Analysis** section of the Environments dialog:

- **Cell Size**: dropdown to select Maximum of Inputs, Minimum of Inputs, or a specific layer
- **Cell Size Projection Method**: how cell size is adjusted during reprojection (usually leave at default)
- **Mask**: an optional raster or feature class defining the active area
- **Snap Raster**: a reference raster for cell alignment

Under the **Processing Extent** section:

- **Extent**: dropdown or manual entry for the geographic bounds of the calculation

In **QGIS**, environment-equivalent settings are configured within individual Processing tools or through the Raster Calculator's output extent and resolution fields. QGIS does not have a centralized environment settings system equivalent to ArcGIS Pro's, but each tool provides parameters for output extent, cell size, and reference layer.

### 6.5.3 The Interaction of Environment Settings

These environment parameters interact in a specific sequence. When you execute a raster calculator expression, the software:

1. Determines the **output extent** from the environment setting (default: intersection of inputs).
2. Determines the **output cell size** from the environment setting (default: maximum of inputs).
3. Determines the **cell alignment** from the snap raster (if specified) or from the first input raster.
4. Creates an empty output grid with the determined extent, cell size, and alignment.
5. For each input raster that does not match the output geometry, performs **resampling** to the output geometry using nearest-neighbor interpolation (the default resampling method for raster calculations).
6. Applies the **analysis mask** to restrict the calculation to the active area.
7. Evaluates the algebraic expression cell by cell.
8. Writes the output.

Understanding this sequence is essential for diagnosing unexpected results. If your output is coarser than expected, check the cell size setting. If it covers a smaller area than expected, check the extent setting. If cell values do not match what you expect from the inputs, check whether resampling has altered the values. If some cells are unexpectedly NODATA, check the mask and the NODATA handling.

## 6.6 Resampling Methods in Detail

### 6.6.1 Nearest-Neighbor Resampling

Nearest-neighbor is the simplest resampling method. For each cell in the target grid, the algorithm finds the cell center in the source grid that is geographically closest to the target cell center, and assigns that value to the target cell. No interpolation or averaging occurs -- the output value is always an exact value from the input grid.

**Advantages**: It is fast, it preserves the original data values exactly, and it works for all data types including categorical data (land cover classes, soil types) where averaging would be meaningless. If your input grid contains land cover class 42 (deciduous forest) and class 11 (open water), nearest-neighbor resampling will assign one of these existing classes to each output cell. It will never produce class 26.5 (a nonsensical average of forest and water).

**Disadvantages**: For continuous data, nearest-neighbor resampling creates a blocky, staircase-like pattern because each output cell receives a single input value with no smooth transition between cells. When resampling to a coarser resolution, many input values are discarded entirely -- only the single nearest value at each output cell center is retained. When resampling to a finer resolution, the same input value is repeated across multiple output cells, creating the appearance of resolution without adding information.

**When to use it**: For categorical or integer data (land cover, soil type, geological unit, flow direction codes, any classified raster). Also appropriate when you need to preserve exact values, such as when resampling a raster of station IDs or zone numbers.

### 6.6.2 Bilinear Interpolation

Bilinear interpolation estimates the value at a target cell center by fitting linear functions in the x and y directions using the four nearest cell centers from the source grid. The algorithm is:

1. Identify the four source cells surrounding the target location (a 2x2 neighborhood).
2. Interpolate linearly in the x-direction along the top edge of this neighborhood.
3. Interpolate linearly in the x-direction along the bottom edge.
4. Interpolate linearly in the y-direction between the two intermediate values from steps 2 and 3.

The mathematical formulation, for a target point at fractional position $(dx, dy)$ within the four surrounding source cells with values $z_{11}, z_{12}, z_{21}, z_{22}$, is:

$$z = z_{11}(1-dx)(1-dy) + z_{21}(dx)(1-dy) + z_{12}(1-dx)(dy) + z_{22}(dx)(dy)$$

where $dx$ and $dy$ range from 0 to 1 and represent the fractional distance from the upper-left corner of the 2x2 neighborhood.

**Advantages**: Produces smooth, continuous output values that transition gradually between source cell values. Does not create the blocky artifacts of nearest-neighbor resampling. Values are weighted averages of the four surrounding cells, providing a better estimate for continuous phenomena.

**Disadvantages**: Output values may not exactly match any input value. The method can slightly smooth sharp gradients. Not appropriate for categorical data (averaging land cover classes is physically meaningless). Slightly slower than nearest-neighbor, though the difference is negligible on modern hardware.

**When to use it**: For continuous floating-point data such as elevation, precipitation, temperature, soil moisture, evapotranspiration, or any other quantity where intermediate values are physically meaningful. This aligns with the spatial-average interpretation of cell values discussed in Chapter 5, Section 5.3.

### 6.6.3 Cubic Convolution

Cubic convolution extends the principle of bilinear interpolation by fitting a cubic (third-degree) polynomial surface through a 4x4 neighborhood of source cells surrounding the target location. This produces smoother results than bilinear interpolation, with better preservation of local gradients and features.

**Advantages**: Smoothest output among the three standard methods. Better preserves sharp features and gradients. Often preferred for visualization purposes.

**Disadvantages**: Can produce output values slightly outside the range of the input data (overshoot). This is problematic for data that has physical bounds -- if precipitation values range from 0 to 100 mm, cubic convolution might produce a value of -2 mm or 102 mm at locations near steep gradients. Requires a 4x4 neighborhood, so it cannot be applied near the edges of the raster. Slowest of the three standard methods.

**When to use it**: For continuous data where visual smoothness is important and where small overshoots are acceptable. Often used for display-quality terrain surfaces. Avoid for data with strict physical bounds unless you apply a post-processing step to clamp values.

### 6.6.4 Majority Resampling

Majority resampling assigns to each output cell the most frequently occurring value among the source cells that fall within its footprint. This is a specialized method designed for categorical data when resampling to a coarser resolution.

**When to use it**: For coarsening categorical rasters (e.g., converting a 10 m land cover grid to 30 m). Nearest-neighbor selects a single point value; majority considers the entire cell area and assigns the dominant category, which is often more representative.

**Table 6.4**: Summary of resampling methods and their appropriate applications

| Method | Type of Data | Preserves Original Values? | Output Smoothness | Speed | Use Case Examples |
|--------|-------------|---------------------------|-------------------|-------|-------------------|
| Nearest Neighbor | Categorical, integer | Yes | Blocky | Fastest | Land cover, soil type, flow direction, zone IDs |
| Bilinear | Continuous, floating-point | No (weighted average) | Smooth | Fast | Precipitation, elevation, temperature, soil moisture |
| Cubic Convolution | Continuous, floating-point | No (may overshoot) | Smoothest | Moderate | Display-quality surfaces, visualization |
| Majority | Categorical | Yes (most common value) | N/A | Moderate | Coarsening land cover, geological maps |

### 6.6.5 Worked Example: Bilinear Resampling of the Precipitation Grid

Let us return to our worked example and apply bilinear interpolation to resample the precipitation grid from 150 m to 100 m, so it matches the infiltration grid. The original precipitation grid (150 m) contains:

```
4     6
3.1   4
```

with cell centers at:

- (75, 225) = 4.0
- (225, 225) = 6.0
- (75, 75) = 3.1
- (225, 75) = 4.0

The target grid (100 m) has cell centers at:

- (50, 250), (150, 250), (250, 250)   -- top row
- (50, 150), (150, 150), (250, 150)   -- middle row
- (50, 50), (150, 50), (250, 50)       -- bottom row

For each target cell, bilinear interpolation uses the four surrounding source cell centers to compute a weighted average. Let us compute each of the nine output values.

**Top row:**

*Cell (1,1) at (50, 250):* This point is at the upper-left corner of the source grid, near cell center (75, 225). It falls outside the convex hull of the four source cell centers, so bilinear interpolation uses the nearest available value. The result is **4.0**.

*Cell (1,2) at (150, 250):* This point lies between the two upper source cells. The fractional x-position between (75, 225) and (225, 225) is $(150 - 75)/(225 - 75) = 0.5$. The y-position is at the upper edge. Interpolating linearly: $4.0 \times (1 - 0.5) + 6.0 \times 0.5 = 5.0$. The result is **5.0**.

*Cell (1,3) at (250, 250):* This point is at the upper-right corner, near cell center (225, 225). The result is **6.0**.

**Middle row:**

*Cell (2,1) at (50, 150):* This point lies between the two left source cells vertically. The fractional y-position between (75, 225) and (75, 75) is $(225 - 150)/(225 - 75) = 0.5$. Interpolating: $4.0 \times (1 - 0.5) + 3.1 \times 0.5 = 3.55$. The result is **3.55**.

*Cell (2,2) at (150, 150):* This point lies at the center of all four source cells. Bilinear interpolation with $dx = 0.5$, $dy = 0.5$:

$$z = 4.0 \times 0.5 \times 0.5 + 6.0 \times 0.5 \times 0.5 + 3.1 \times 0.5 \times 0.5 + 4.0 \times 0.5 \times 0.5$$
$$z = 1.0 + 1.5 + 0.775 + 1.0 = 4.275$$

The result is **4.275**.

*Cell (2,3) at (250, 150):* Symmetric to cell (2,1) but on the right side. Interpolating between 6.0 and 4.0: $6.0 \times 0.5 + 4.0 \times 0.5 = 5.0$. The result is **5.0**.

**Bottom row:**

*Cell (3,1) at (50, 50):* Near the lower-left source cell center. The result is **3.1**.

*Cell (3,2) at (150, 50):* Between the two lower source cells. Interpolating: $3.1 \times 0.5 + 4.0 \times 0.5 = 3.55$. The result is **3.55**.

*Cell (3,3) at (250, 50):* Near the lower-right source cell center. The result is **4.0**.

The resampled precipitation grid at 100 m is therefore:

**Table 6.5**: Bilinear-resampled precipitation grid (100 m resolution)

| | Col 1 | Col 2 | Col 3 |
|---|---|---|---|
| **Row 1** | 4.0 | 5.0 | 6.0 |
| **Row 2** | 3.55 | 4.275 | 5.0 |
| **Row 3** | 3.1 | 3.55 | 4.0 |

Notice how the interpolated values create smooth spatial transitions. The value 5.0 in the top-center cell is midway between the 4.0 and 6.0 original values. The value 4.275 at the center of the grid is the average of all four original values (weighted equally because the center point is equidistant from all four source centers). The value 3.55 appears in two locations, representing the midpoint between 3.1 and 4.0 in different directions.

| (1,2) | 5.0 | 3.0 | 2.00 |
| (1,3) | 6.0 | 4.0 | 2.00 |
| (2,1) | 3.55 | 1.8 | 1.75 |
| (2,2) | 4.275 | 2.5 | 1.775 |
| (2,3) | 5.0 | 3.5 | 1.50 |
| (3,1) | 3.1 | 1.5 | 1.60 |
| (3,2) | 3.55 | 2.0 | 1.55 |

Compare this with the default-behavior result from Section 6.4.3:

```
1.8   2.0
1.6   1.3
```

The bilinear-resampled result has three times as many cells, uses all nine infiltration values (none discarded), and provides smoother spatial transitions in the runoff pattern. The key difference is not in the extremes but in the interior: the bilinear result reveals that the center of the study area has a runoff of 1.775 mm -- an intermediate value that the coarse default result completely misses.

Whether this additional detail matters depends on the application. For a watershed-average runoff calculation, the difference may be negligible. For a detailed urban flood routing model that needs cell-level runoff estimates, the difference is critical.

### 6.6.7 Comparing Nearest-Neighbor and Bilinear Results

It is instructive to compare the nearest-neighbor and bilinear resampling approaches side by side, because the comparison illustrates a fundamental trade-off in raster analysis.

If we had instead resampled the precipitation grid to 100 m using nearest-neighbor (for comparison with the bilinear result above), each target cell would receive the value of the single nearest source cell:

**Table 6.7**: Nearest-neighbor resampled precipitation grid (100 m resolution)

| | Col 1 | Col 2 | Col 3 |
|---|---|---|---|
| **Row 1** | 4.0 | 4.0 | 6.0 |
| **Row 2** | 4.0 | 4.0 | 6.0 |
| **Row 3** | 3.1 | 3.1 | 4.0 |

Notice the blocky pattern: the value 4.0 fills the upper-left quadrant (four cells), 6.0 fills the upper-right column, 3.1 fills the lower-left corner, and 4.0 appears in the lower-right. There are no intermediate values. Each output cell simply repeats the nearest source value.

The resulting runoff grid using nearest-neighbor resampled precipitation:

| Cell Position | NN Precipitation | Infiltration | NN Runoff | Bilinear Runoff | Difference |
|--------------|-----------------|-------------|-----------|----------------|------------|
| (1,1) | 4.0 | 2.2 | 1.80 | 1.80 | 0.00 |
| (1,2) | 4.0 | 3.0 | 1.00 | 2.00 | -1.00 |
| (1,3) | 6.0 | 4.0 | 2.00 | 2.00 | 0.00 |
| (2,1) | 4.0 | 1.8 | 2.20 | 1.75 | +0.45 |
| (2,2) | 4.0 | 2.5 | 1.50 | 1.775 | -0.275 |
| (2,3) | 6.0 | 3.5 | 2.50 | 1.50 | +1.00 |
| (3,1) | 3.1 | 1.5 | 1.60 | 1.60 | 0.00 |

The differences are striking. At cell (1,2), the nearest-neighbor method assigns precipitation = 4.0 (the value from the cell to the left), while bilinear assigns 5.0 (the average of 4.0 and 6.0). This produces a runoff difference of 1.0 mm -- a 50% discrepancy at that cell. At cell (2,3), the nearest-neighbor assigns 6.0 while bilinear assigns 5.0, producing a 1.0 mm difference in the opposite direction.

The take-away is clear: for continuous data like precipitation, bilinear interpolation produces more physically realistic results because it creates smooth spatial transitions rather than abrupt block boundaries. The nearest-neighbor result implies that precipitation jumps discontinuously from 4.0 to 6.0 at a cell boundary, which is physically implausible for a smoothly varying atmospheric quantity.

## 6.7 NODATA Handling

### 6.7.1 What NODATA Means

Every raster format supports a designated NODATA value -- a sentinel number that indicates "no valid data exists at this cell." Common NODATA values include -9999, -3.4028235e+38 (the negative of the maximum 32-bit float), or specific values defined in the raster header. NODATA is distinct from zero: a cell with zero precipitation means "it did not rain here," while a cell with NODATA precipitation means "we have no information about whether it rained here."

NODATA arises for several reasons in hydrological raster data. The raster may cover a rectangular extent but the actual data covers an irregular region (a watershed boundary, a country border, or a satellite swath), leaving cells outside the data region as NODATA. The source data may have gaps -- a rain gauge network interpolated to a grid will have less certainty (and sometimes explicit gaps) in areas far from any gauge. Sensor failures, cloud contamination in satellite imagery, or quality control flags can all produce NODATA cells.

### 6.7.2 The Propagation Rule

The fundamental rule of NODATA in map algebra is: **if any input cell is NODATA, the corresponding output cell is NODATA.** This rule applies regardless of the operation. Adding a valid value to NODATA produces NODATA. Multiplying NODATA by zero produces NODATA (even though mathematically anything times zero is zero). Comparing NODATA to a threshold produces NODATA.

This behavior is both a safety feature and a potential problem. It is safe because it prevents the software from computing a result based on incomplete information -- if you do not know the infiltration at a cell, you should not claim to know the runoff. But it can be problematic because NODATA propagation is aggressive: a single NODATA cell in any one of a dozen input layers will create a NODATA cell in the output, even if all other inputs are valid. In complex multi-layer calculations, NODATA areas can expand rapidly, leaving much less valid output than you expected.

### 6.7.3 Division by Zero

A special case of NODATA generation is division by zero. If you write the raster calculator expression `"grid_a" / "grid_b"` and any cell in grid_b has a value of zero, the output at that cell will be NODATA (or will generate an error, depending on the software). This behavior is mathematically correct -- division by zero is undefined -- but it has a practical consequence that can be exploited.

### 6.7.4 NODATA as a Masking Tool

The division-by-zero behavior provides a simple masking technique. Suppose you want to restrict a runoff grid to only those cells within a particular watershed. Create a binary mask raster where cells inside the watershed have value 1 and cells outside have value 0. Then divide the runoff grid by this mask:

```
"runoff" / "watershed_mask"
```

Inside the watershed, each runoff value is divided by 1, producing the original runoff value. Outside the watershed, each runoff value is divided by 0, producing NODATA. The result is a runoff grid that exists only within the watershed boundary, with NODATA everywhere else.

An alternative, often cleaner approach in ArcGIS Pro uses the `SetNull` or `Con` functions:

```python
SetNull("watershed_mask" == 0, "runoff")
```

This explicitly sets cells to Null (NODATA) where the mask is 0 and retains the runoff value everywhere else. In QGIS, the equivalent can be achieved with a conditional expression in the raster calculator.

### 6.7.5 Strategies for Managing NODATA

When NODATA propagation threatens to reduce your output to an unacceptably small area, several strategies are available:

**Fill NODATA cells before calculation.** If NODATA cells represent small gaps in otherwise continuous data, you can fill them using interpolation (the GDAL `fillnodata` algorithm, ArcGIS's Raster Interpolation tools, or QGIS's GRASS `r.fillnulls`). This is appropriate when the gaps are small and the surrounding data provides reasonable information for interpolation. It is not appropriate when NODATA represents genuinely absent information, such as cells outside a country boundary.

**Use conditional logic to substitute values.** The `Con` function can replace NODATA with a default value:

```python
Con(IsNull("infiltration"), 0, "infiltration")
```

This replaces NODATA infiltration cells with zero, meaning those cells will generate maximum runoff (all precipitation becomes runoff). Whether this is physically appropriate depends on why the data is missing.

**Perform calculations in stages.** If you have a dozen inputs and only one has significant NODATA areas, perform the map algebra in stages: first compute intermediate results using the inputs that have complete coverage, then combine those intermediates with the incomplete input. This isolates the NODATA propagation to the final step and may preserve more valid output area.

**Accept the NODATA and document it.** Sometimes the honest answer is that you do not have data everywhere, and the output should reflect that. A runoff map with NODATA cells in areas of missing infiltration data is more honest than a runoff map with fabricated values in those cells.

## 6.8 Practical Workflow for Raster Calculations

### 6.8.1 Before You Calculate: The Inspection Step

The most common source of errors in raster calculations is not the algebra -- it is the failure to inspect the inputs before calculating. A disciplined workflow begins with inspection, proceeds through reconciliation, and only then arrives at the calculation itself.

For every input raster, verify the following properties (in ArcGIS Pro, right-click the layer and open Properties; in QGIS, open the Layer Properties dialog):

**Coordinate system.** Are all inputs in the same coordinate system? If one raster is in WGS 84 geographic coordinates (degrees) and another is in UTM Zone 34N (meters), the cell sizes are in different units and cannot be directly compared. All inputs must be projected to a common coordinate system before raster calculations. For Hungary, the common choices are HD72/EOV (EPSG:23700) for national datasets, ETRS89/TM34 (EPSG:25834) for EU-wide integration, or WGS 84/UTM Zone 34N (EPSG:32634) for general use.

**Cell size.** What is the cell size of each input? Are they the same? If not, which is the finest, which is the coarsest, and what output resolution do you want?

**Extent.** Do all inputs cover the same geographic area? If not, is the analysis extent the intersection (only where all inputs overlap) or do you need a different extent?

**Data type.** Is each input integer or floating-point? If you are going to perform arithmetic that produces fractional results, you need floating-point output regardless of whether the inputs are integers.

**NODATA.** Does each input have NODATA cells? Where are they? Do the NODATA areas of different inputs overlap, or will NODATA propagation significantly reduce the valid output area?

**Value range.** Are the values in each input within expected ranges? A precipitation raster with negative values or a soil permeability raster with values of -9999 (the NODATA sentinel accidentally included as a valid value) will produce garbage results.

**Table 6.8**: Input inspection checklist for raster calculations

| Property | What to Check | Common Problems |
|----------|---------------|-----------------|
| Coordinate system | Same for all inputs | Mismatch produces wrong overlap or wildly wrong values |
| Cell size | Same or reconciliation planned | Unintended coarsening of output |
| Extent | Overlap sufficient for analysis area | Missing data at edges of study area |
| Data type | Integer vs. float | Truncation of decimal values in output |
| NODATA value | Defined and appropriate | Sentinel value (-9999) treated as data |
| Value range | Physically reasonable | Corrupted data, wrong units, NODATA treated as valid |
| Number of bands | Single-band for each variable | Accidentally using multi-band raster (only band 1 used) |

### 6.8.2 Reconciliation: Bringing Inputs into Alignment

Once you have inspected all inputs, reconcile any mismatches before performing the calculation. The recommended sequence is:

**Step 1: Project to a common coordinate system.** If inputs are in different coordinate systems, use the Project Raster tool (ArcGIS Pro: Data Management > Projections and Transformations > Raster > Project Raster; QGIS: Processing > GDAL > Raster projections > Warp) to bring all inputs to the same coordinate system. Choose a projected coordinate system (meters or feet) rather than a geographic coordinate system (degrees) for hydrological calculations, because cell sizes and areas need to be in linear units.

**Step 2: Resample to a common cell size.** Use the Resample tool to bring all inputs to the desired output cell size. Choose the resampling method appropriate for each data type (bilinear for continuous data, nearest-neighbor for categorical data). Set the target cell size to match the finest relevant input, or to match a reference grid.

**Step 3: Snap to a reference grid.** Set the snap raster in the analysis environment to ensure all resampled grids share the same cell alignment. Even if two grids have identical cell sizes, they may be offset by a fraction of a cell width, causing subtle misalignment that the Raster Calculator must resolve through additional resampling at calculation time.

**Step 4: Verify alignment.** After reconciliation, open the resampled grids in the GIS and visually confirm that they overlay correctly. Use the Explore tool to check that values at specific locations match expectations.

### 6.8.3 The Calculation Itself

With reconciled inputs, the raster calculation is usually the easiest step. Enter the expression in the Raster Calculator, specify the output raster name (with `.tif` extension for GeoTIFF), and execute. For complex expressions, build them incrementally: first compute intermediate results, verify them, and then combine the intermediates. This approach makes it easier to identify which step introduced any errors.

### 6.8.4 After You Calculate: Verification

Never trust a raster calculation output without verification. At minimum:

**Spot-check individual cell values.** Use the Explore tool to click on several locations and verify that the output value equals the expected result of the expression applied to the input values at that location. Check cells in the interior, near edges, and near NODATA boundaries.

**Check the value range.** Open the layer properties and examine the minimum, maximum, mean, and standard deviation of the output. Are these physically reasonable? If you computed runoff as precipitation minus infiltration, the runoff should be non-negative everywhere (if you used a Con expression to enforce this) and should not exceed the maximum precipitation.

**Check the spatial pattern.** Visually examine the output map. Does the pattern make sense? If precipitation is highest in the northeast and infiltration is highest in the southeast, runoff should be highest in areas of high precipitation and low infiltration. A pattern that contradicts this expectation signals an error.

**Compare with known values.** If you have independent data -- gauged runoff at a watershed outlet, for example -- compare the spatially averaged raster result with the known value. This does not validate every cell, but it provides a bulk sanity check.

## 6.9 Beyond Simple Arithmetic: Types of Local Operations

### 6.9.1 Arithmetic Operations

The most common map algebra operations are the basic arithmetic operators: addition, subtraction, multiplication, and division. These are used to implement physical equations (precipitation minus infiltration), unit conversions (multiplying a grid by a constant to convert from inches to millimeters), and scaling operations (dividing by a normalizing factor).

Exponentiation is also common. The SCS Curve Number equation for runoff involves squaring a quantity:

$$Q = \frac{(P - I_a)^2}{P - I_a + S}$$

In raster calculator syntax (ArcGIS Pro):

```python
Power("precipitation" - "initial_abstraction", 2) / ("precipitation" - "initial_abstraction" + "retention")
```

or using the `**` operator:

```python
("precipitation" - "initial_abstraction") ** 2 / ("precipitation" - "initial_abstraction" + "retention")
```

### 6.9.2 Comparison and Logical Operations

Map algebra supports comparison operators that produce boolean (true/false or 1/0) output grids:

- Greater than: `"elevation" > 500` produces 1 where elevation exceeds 500 m, 0 elsewhere
- Less than or equal: `"slope" <= 5` identifies flat areas
- Equal to: `"landcover" == 42` selects cells with a specific land cover class
- Not equal: `"soiltype" != -9999` identifies cells with valid data

Boolean grids can be combined with logical operators (AND, OR, NOT) to create complex selection criteria:

```python
("slope" > 10) & ("soil_perm" < 5) & ("landcover" == 21)
```

This identifies cells that are steep, have low-permeability soils, and are covered by developed land -- a high-risk combination for surface runoff.

### 6.9.3 Mathematical Functions

Most raster calculator implementations support standard mathematical functions applied cell-by-cell:

- **Abs**: absolute value
- **Sqrt**: square root
- **Log, Log10, Log2**: natural, base-10, and base-2 logarithms
- **Exp**: exponential ($e^x$)
- **Sin, Cos, Tan**: trigonometric functions (inputs in radians)
- **Int, Float**: type conversion (integer to float or vice versa)
- **Round, Floor, Ceil**: rounding operations

These functions enable the implementation of virtually any physically-based equation as a map algebra expression. Manning's equation for flow velocity, for example, involves a fractional exponent:

$$V = \frac{1}{n} R^{2/3} S^{1/2}$$

where $V$ is velocity, $n$ is Manning's roughness coefficient, $R$ is hydraulic radius, and $S$ is slope. If each of these terms is available as a raster, the velocity grid can be computed as:

```python
(1.0 / "mannings_n") * Power("hydraulic_radius", 0.667) * SquareRoot("slope")
```

### 6.9.4 Statistical Operations Across Multiple Grids

A powerful category of local operations computes cell-by-cell statistics across a stack of raster grids. The Cell Statistics tool (ArcGIS Pro: Spatial Analyst > Local) takes multiple input rasters and computes, at each cell, the mean, maximum, minimum, range, standard deviation, sum, variety, or majority across the inputs.

This is invaluable for climate and hydrological analysis. If you have 30 years of annual precipitation grids, Cell Statistics can compute the long-term mean precipitation at each cell, the wettest year at each cell, the driest year, the standard deviation (indicating year-to-year variability), and the range. Similarly, if you have precipitation forecasts from 10 different climate models, Cell Statistics can compute the ensemble mean and spread at each cell, providing both a best-estimate prediction and a measure of model uncertainty.

In QGIS, equivalent functionality is available through the GRASS tool `r.series` and through the Processing framework's raster layer statistics tools.

### 6.10.2 A Hungarian Example: Estimating Runoff in the Zala River Catchment

To illustrate map algebra in a European context, consider estimating excess precipitation for the Zala River catchment in western Hungary. The Zala is the largest tributary of Lake Balaton, Hungary's most important surface water body, and its water quality and runoff dynamics are of significant national interest.

The input grids might include:

- **Precipitation**: OMSZ (Hungarian Meteorological Service) radar composite, 1 km resolution, in HD72/EOV projection (EPSG:23700), providing event rainfall totals.
- **Soil hydrologic properties**: Derived from the AGROTOPO digital soil map of Hungary, originally at 1:100,000 scale, rasterized at 100 m resolution, classifying soils by water retention and permeability categories.
- **Land cover**: CORINE Land Cover from Copernicus, 100 m resolution, in ETRS89/LAEA projection (EPSG:3035).

Before any map algebra can proceed, three reconciliation issues must be addressed:

1. **Coordinate system mismatch**: The precipitation grid is in EOV, the land cover is in ETRS89/LAEA. Both must be projected to a common system (EOV is the natural choice for a national analysis in Hungary).
2. **Cell size mismatch**: 1 km precipitation, 100 m soil and land cover. The target resolution depends on the application -- 100 m for a detailed analysis, 1 km for a quick regional estimate.
3. **Extent differences**: The precipitation grid covers all of Hungary, the soil map covers Hungary, and the CORINE land cover covers all of Europe. The analysis extent should be clipped to the Zala catchment boundary.

The workflow follows the same steps outlined in Section 6.8: project to common coordinate system, resample to common cell size (bilinear for precipitation, nearest-neighbor for the categorical soil and land cover grids), set the analysis mask to the Zala catchment boundary, and perform the map algebra calculation. The mathematics is identical to the simple example -- only the data sources and scale differ.

### 6.10.3 Cloud-Based Map Algebra

The traditional workflow described in this chapter -- download data, load into desktop GIS, reconcile, calculate -- works well for local-scale analyses with a few input layers. But when the analysis spans continental or global extents, or involves time series of hundreds of raster grids, the data volumes become impractical for desktop processing.

Cloud-based platforms address this by bringing the computation to the data. **Google Earth Engine** stores a multi-petabyte catalog of satellite imagery, climate data, terrain models, and derived products, and provides a complete map algebra system that operates on these data without downloading. An analyst can write an expression in JavaScript or Python that computes, say, the 20-year mean annual NDVI minus a drought-year NDVI for every 30 m cell across the entire Danube basin, and the computation runs on Google's servers in minutes.

**Microsoft Planetary Computer** offers a similar capability, with a particular focus on the Copernicus data catalog and integration with the Python scientific ecosystem (xarray, dask, rasterio). The **Copernicus Data Space Ecosystem** operated by the European Space Agency provides direct access to Sentinel satellite data and derived products, with processing capabilities through JupyterHub environments.

For the water professional, the key insight is that map algebra is not limited to the Raster Calculator dialog in a desktop GIS. The same mathematical principles -- cell-by-cell computation, resampling for reconciliation, NODATA propagation -- apply in cloud environments. What changes is the scale: instead of processing a single watershed, you can process an entire continent. The conceptual foundation from this chapter transfers directly.

## 6.11 Common Pitfalls and How to Avoid Them

### 6.11.1 The Silent Resampling Problem

The most insidious pitfall in raster calculations is that resampling happens silently. When you enter an expression in the Raster Calculator with mismatched inputs, the software does not warn you that it is resampling one or more inputs. It does not tell you which resampling method it used, or that it coarsened your expensive high-resolution data to match a coarser grid. It simply produces a result, and unless you inspect it carefully, you may never realize that the output does not reflect the full information content of your inputs.

The defense is the inspection-first workflow described in Section 6.8. Check cell sizes before calculating. If they differ, make a conscious decision about the target resolution and perform explicit resampling with an appropriate method. Do not rely on the defaults.

### 6.11.2 The False Resolution Problem

Resampling a 1 km precipitation grid to 30 m does not create 30 m precipitation data. It creates a 30 m grid whose values are smooth interpolations of 1 km data. The spatial information content remains at 1 km -- the finer grid is cosmetically detailed but informationally unchanged. This matters when interpreting results: a runoff map computed from 30 m land cover and 30 m soil data combined with resampled-to-30 m precipitation looks like it has 30 m resolution, but the precipitation component is smoothly varying at 1 km, and the runoff pattern at scales finer than 1 km is driven entirely by the soil and land cover variations.

This is not necessarily wrong -- it may be exactly what you want for a distributed model. But you must understand and document it, because a reader seeing a 30 m runoff map might assume all inputs contributed information at 30 m.

### 6.11.3 The Extent Mismatch Problem

If the analysis extent defaults to the intersection of inputs and one input covers a much smaller area than the others, the output will be restricted to that small area. This can be confusing when you expected a watershed-scale result but got only a small patch because one input raster had a limited extent.

Conversely, if you set the extent to the union of inputs, cells where any input is missing will become NODATA through the propagation rule, potentially leaving you with even less valid area than the intersection.

The solution is to plan the extent deliberately: define it as the watershed boundary (using the analysis mask), verify that all inputs cover the entire watershed, and address any gaps before running the calculation.

### 6.11.4 The Integer Truncation Problem

If one input is an integer raster and the calculation produces fractional results, some software configurations may store the output as an integer, truncating the decimal portion. In ArcGIS Pro, this is generally handled automatically (the output is promoted to float if any operation produces fractional results), but in QGIS and especially in scripted workflows, you may need to explicitly specify a floating-point output type.

Always check the output data type after calculation. If you expected floating-point values and got integers, the results are wrong and need to be recomputed with the correct output type.

### 6.11.5 The Coordinate System Trap

Performing raster calculations on grids in geographic coordinates (latitude/longitude in degrees) is technically possible but often inappropriate for hydrological analysis. In geographic coordinates, cell sizes vary with latitude -- a "0.001 degree" cell is approximately 111 m north-south but only about 75 m east-west at 47 degrees latitude (the latitude of central Hungary). This means that cells are not square, areas cannot be directly computed from cell counts, and distance-dependent operations produce distorted results.

For any analysis that involves spatial quantities with physical units (precipitation in mm over an area, runoff volumes in cubic meters, slopes in degrees), project all inputs to a metric coordinate system before computing. The coordinate system reconciliation should be the very first step in the workflow, before any cell size reconciliation or map algebra.

## 6.12 From Raster Calculator to Distributed Modeling

### 6.12.1 The Building Blocks

Everything in this chapter -- map algebra expressions, cell size reconciliation, resampling, NODATA handling -- constitutes the building blocks of spatially distributed hydrological modeling. A distributed model is, at its core, a sequence of map algebra operations applied to grids that represent the spatial variability of the landscape and the atmosphere.

The progression through the next several chapters follows this logic:

- **Chapter 7** applies focal (neighborhood) operations to elevation grids to compute slope and aspect -- spatial derivatives that control the direction and velocity of water movement.
- **Chapter 8** uses spatial interpolation to create precipitation grids from point measurements -- converting sparse gauge observations into continuous raster surfaces suitable for map algebra.
- **Chapters 9-12** build complete hydrological workflows: filling depressions in DEMs, computing flow directions, accumulating flow to identify streams, and delineating watersheds. Each of these workflows is a chain of raster operations, and each operation follows the same principles of cell-by-cell computation, environment control, and NODATA handling described here.

The raster calculator is where you start. By the time you reach Chapter 12, you will be chaining dozens of raster operations into automated workflows that transform raw elevation data and weather observations into flood forecasts. But the foundation -- the idea that you can write an equation and apply it to a million grid cells at once -- is what you learned in this chapter.

### 6.12.2 The Scale Question

A persistent question in distributed modeling is: at what cell size should the model run? The answer depends on the application, the available data, and the computational resources, but the principles from this chapter provide the framework for thinking about it.

The cell size should not be finer than the finest information that actually varies at that scale. If your precipitation data is at 1 km and your soil data is at 250 m, running the model at 30 m adds resolution from only the land cover and DEM components -- the precipitation and soil fields are merely resampled, not refined. Whether this matters depends on how sensitive the output is to the fine-scale variations in land cover and topography versus the coarser-scale variations in precipitation and soils. In steep mountainous terrain with fine-scale land use patterns, the 30 m resolution may capture important variability. In a flat agricultural plain with homogeneous cropping, 250 m may be perfectly adequate.

The computational cost scales roughly with the number of cells, which scales as the inverse square of the cell size. Halving the cell size quadruples the number of cells and roughly quadruples the computation time and storage. Moving from 100 m to 10 m increases the cell count by a factor of 100. For a single map algebra operation, this is negligible on modern hardware. For a multi-step hydrological simulation running at hourly time steps over a 30-year period, the difference between 100 m and 10 m can be the difference between minutes and days of computation.

These numbers explain why cell size reconciliation is not just a technical detail but a design decision with practical consequences for every distributed modeling project.

## 6.13 Putting It All Together: A Step-by-Step Workflow

To consolidate everything in this chapter, here is a complete step-by-step workflow for performing a raster calculation correctly, from data inspection to result verification. This workflow applies regardless of the specific formula, the GIS software, or the geographic context.

**Step 1: Define the question.** What physical quantity are you computing? What formula will you use? What inputs do you need?

**Step 2: Assemble the inputs.** Gather all required raster datasets. Document their sources, original resolutions, coordinate systems, and any known quality issues.

**Step 3: Inspect every input.** For each raster, verify: coordinate system, cell size, extent, data type, NODATA definition, value range, number of bands. Record this information in a table (Table 6.8 provides the template).

**Step 4: Choose the target geometry.** Decide on the output coordinate system, cell size, and extent. The coordinate system should be a projected system in linear units. The cell size should reflect the finest resolution at which the analysis is meaningful. The extent should match the study area (typically a watershed boundary).

**Step 5: Reconcile the inputs.** Project all inputs to the common coordinate system. Resample to the common cell size using the appropriate method for each data type. Set the snap raster. Clip to the analysis extent if needed to manage file sizes.

**Step 6: Verify reconciliation.** Open all reconciled inputs in the GIS. Confirm that they overlay correctly. Spot-check values to ensure resampling produced reasonable results.

**Step 7: Perform the calculation.** Enter the expression in the Raster Calculator (or execute the script). For complex formulas, build in stages and verify each intermediate result.

**Step 8: Verify the output.** Check value range, spatial pattern, individual cell values, and comparison with known reference data. Document any discrepancies.

**Step 9: Document everything.** Record the formula, the input datasets and their sources, the reconciliation choices (target cell size, resampling methods, snap raster), the environment settings, and the verification results. Future-you (and your colleagues) will thank present-you.

This workflow may seem laborious for a simple subtraction of two grids. But raster calculations in practice are rarely simple: they involve multiple inputs at different resolutions, complex formulas with conditional logic, and results that feed into subsequent analyses. The disciplined approach outlined here prevents the subtle errors that are easy to make and hard to detect in raster-based spatial analysis.
